## 💡 스파크 역할 리마인드

### 1. 드라이버 (Driver): 전체 작업의 지휘자

드라이버는 사용자가 작성한 코드가 실행되는 메인 프로세스로 단순히 계획만 세우는 것이 아니라, 작업이 끝날 때까지 전체 과정을 총괄

- **실행 계획 수립:** 사용자의 코드를 분석하여 DAG(Directed Acyclic Graph) 생성, 이를 실제 실행 가능한 최소 단위인 태스크(Task)로 쪼갬
- **태스크 스케줄링 및 배포:** 쪼개진 태스크들을 어느 엑시큐터에서 실행할지 결정하고, 네트워크를 통해 태스크를 전달
- **상태 추적:** 어떤 태스크가 성공했는지, 실패해서 재실행이 필요한지를 실시간으로 모니터링

### 2. 클러스터 매니저 (Cluster Manager): 리소스 관리자

클러스터 매니저(YARN, Kubernetes, Standalone 등)는 물리적 자원(CPU, 메모리)만 관리하며 스파크 내부의 구체적인 로직(데이터 처리 내용)은 모름

- **리소스 할당:** 드라이버가 "엑시큐터 10개가 필요해"라고 요청하면, 클러스터 매니저는 현재 노드들의 여유 공간을 확인하여 자원을 빌려줌
- **엑시큐터 실행:** 할당된 자원(워커 노드) 위에서 엑시큐터 프로세스가 뜰 수 있도록 명령
- **그 외:** 드라이버와 엑시큐터 사이의 구체적인 데이터 통신이나 태스크 분배에는 관여하지 않음

### 3. 워커 노드와 엑시큐터 (Worker & Executor)

- **워커 노드(Worker):** 실제 물리적인 서버 장비
- **엑시큐터(Executor):**
    - 워커 노드 안에서 실행되는 실제 계산 프로세스
    - 드라이버가 준 태스크를 수행하고 데이터를 메모리나 디스크에 저장

### 4. 작업 흐름 재구성 (Timeline)

1. **드라이버**가 실행되면서 **클러스터 매니저**에게 "나 이만큼의 자원이 필요해"라고 요청
2. **클러스터 매니저**는 워커 노드들을 살펴보고 자원을 확보한 뒤, 그 위에서 **엑시큐터**들을 실행
3. 이제 자원이 준비되었으므로, **드라이버**가 직접 **엑시큐터**들에게 "너는 이 데이터를 처리해(Task)"라고 명령
4. **엑시큐터**는 작업을 마치고 결과를 **드라이버**에게 보고

### 1.2.1 정적 할당과 동적 할당

Spark 애플리케이션을 실행할 때 컴퓨터 자원을 어떻게 나누어 줄지 결정하는 방식

1. 정적 리소스 할당 (Static Allocation)

- **방식**: 처음에 사용할 익스큐터(Executor) 개수를 정해놓고 시작
- **특징**: 작업이 많아져서 줄이 길게 서도(Queue) 자원을 더 가져올 수 없음. 반대로 작업이 없어서 놀고 있어도 자원을 붙잡고 있어 낭비가 발생

2. 동적 리소스 할당 (Dynamic Allocation)

- **방식**: 워크로드의 양에 따라 익스큐터를 자동으로 늘리거나 줄임
- **특징**: 갑자기 데이터가 몰리는 스트리밍 작업이나 여러 사람이 함께 쓰는 환경(YARN, Kubernetes 등)에서 매우 유리함. 자원이 남으면 반납하고, 부족하면 새로 요청하는 유연함을 가짐

3. 동적 할당을 위한 주요 설정값

- **spark.dynamicAllocation.enabled**: `true`로 설정하여 기능 활성화
- **minExecutors / maxExecutors**: 최소 몇 대는 유지할지, 최대 몇 대까지 늘릴지 범위를 정함
- **schedulerBacklogTimeout**: 할당되지 못한 작업이 일정 시간(예: 1분) 넘게 대기하면 "자원이 더 필요하구나" 판단하고 새 익스큐터를 요청
- **executorIdleTimeout**: 익스큐터가 할 일 없이 노는 시간(예: 2분)이 지나면 "자원을 반납해"라고 명령하여 종료

### 1.2.2 스파크 이그제큐터의 메모리와 셔플 서비스 설정

1. 스파크 이그제큐터 메모리

`spark.executor.memory`로 설정한 값은 크게 세 부분으로 쪼개짐

- **예약 메모리 (Reserved Memory)**: 시스템 오류(OOM)를 막기 위해 기본적으로 300MB 정도 비워두는 안전장치
- **실행 메모리 (Execution Memory)**: 데이터를 섞고(Shuffle), 합치고(Join), 정렬하는 '연산'에 직접 쓰이는 공간 (보통 60% 비중)
- **저장 메모리 (Storage Memory)**: 계산된 데이터를 나중에 또 쓰려고 저장(Caching)해두는 공간 (보통 40% 비중)
- **핵심 원리**: 실행 메모리와 저장 메모리는 한쪽이 메모리를 안 쓰고 있으면 서로 빌려줄 수 있는 유동적인 구조

2 셔플 서비스 최적화

- 데이터를 재배치하는 **셔플(Shuffle)** 단계는 디스크 읽기/쓰기가 많아 속도가 느려질 수 있음
- 이를 막기 위해 디스크에 쓰기 전 버퍼 메모리를 늘려주거나 전용 셔플 서비스를 튜닝하는 과정이 필요
- 아무리 동적 할당을 잘해도 이 I/O 병목을 해결하지 못하면 성능이 제대로 나오지 않음

### 1.2.3 스파크 병렬성 최대화

1. Spark 병렬성의 기초: 파티션(Partition)

Spark 성능의 핵심은 데이터를 여러 조각으로 나누어 동시에 처리하는 병렬성에 있습니다.

- **파티션의 정의**: 디스크에 저장된 데이터를 설정 가능한 크기(기본 128MB)로 나눈 논리적인 데이터 뭉치.
- **병렬 처리 원리**: Spark는 하나의 태스크(Task)가 하나의 파티션을 처리하도록 설계됨. 따라서 실행기(Executor)의 코어 수만큼 파티션이 존재해야 모든 코어가 쉬지 않고 일함.
- **파티션 생성 방식**:
    - **MaxPartitionBytes**: 읽기 시점에 파티션 크기를 결정. 지나치게 작으면 파일 입출력 오버헤드가 발생하는 'Small File Problem' 발생 가능.
    - **repartition()**: 코드상에서 명시적으로 파티션 개수를 조정.
    - **Shuffle Partitions**: `groupBy()`나 `join()` 같은 넓은 변환(Wide Transformation) 시 생성되며, 기본값은 200개임.

---

2. 데이터 재사용을 위한 캐싱과 영속화

자주 사용하는 데이터프레임은 메모리에 올려두어 재계산 비용을 줄일 수 있습니다.

- **cache() vs persist()**: `cache()`는 기본적으로 메모리에만 저장하며, `persist()`는 저장 수준(Storage Level)을 메모리, 디스크, 직렬화 여부 등에 따라 상세히 설정 가능.
- **저장 수준(Storage Level)**:
    - **MEMORY_ONLY**: 메모리에 객체 형태로 저장.
    - **MEMORY_AND_DISK**: 메모리 부족 시 초과분은 디스크에 저장.
    - **DISK_ONLY**: 디스크에만 저장.
- **주의사항**: 캐싱은 액션(Action, 예: `count()`)이 호출될 때 실제로 수행됨. 무분별한 캐싱은 자원 낭비를 초래하므로 반복 계산이 필요한 경우에만 권장.

3. 효율적인 조인(Join) 전략

데이터 이동(Shuffle)을 최소화하는 것이 성능 최적화의 관건입니다.

### 브로드캐스트 해시 조인 (Broadcast Hash Join, BHJ)

- **작동 방식**: 작은 데이터프레임을 모든 실행기에 복제하여 네트워크를 통한 전체 데이터 이동(Shuffle)을 방지함.
- **조건**: 한쪽 데이터셋이 기본 10MB 이하일 때 자동 적용되나 설정으로 변경 가능.
- **장점**: 셔플이 없어 가장 빠름.

### 셔플 소트 머지 조인 (Shuffle Sort Merge Join, SMJ)

- **작동 방식**: 두 큰 데이터셋을 공통 키로 정렬(Sort)한 뒤 병합(Merge).
- **단점**: 데이터를 재배치하는 과정에서 비용이 많이 드는 '익스체인지(Exchange)' 단계가 발생.
- **최적화(Bucketing)**: 데이터를 미리 특정 키로 분류하여 저장하는 '버켓팅(Bucketing)'을 적용하면 정렬과 셔플 단계를 건너뛰고 바로 병합 가능.

## 1. 셔플 소트 머지 조인(SMJ)의 사용 시점

SMJ는 두 큰 데이터셋을 합칠 때 사용하는 기본 전략입니다. 다음과 같은 상황에서 가장 큰 효과를 발휘합니다.

- **대규모 데이터셋**: 두 데이터셋 모두 메모리에 올리기에는 너무 커서 브로드캐스트 조인이 불가능할 때 사용.
- **정렬 및 해시 가능 여부**: 각 데이터셋의 키가 정렬 가능하고, Spark가 동일한 파티션으로 해싱할 수 있을 때 유리.
- **동등 조인(Equi-joins)**: 두 데이터셋의 키가 일치하는지 비교하는 조인 수행 시 적합.
- **버켓팅(Bucketing) 활용**: 데이터를 미리 정렬하고 버켓팅해 두었다면, 네트워크를 통한 대량 데이터 이동인 익스체인지(Exchange)와 정렬 단계를 생략할 수 있어 성능이 극대화됨.

## 2. Spark UI를 통한 어플리케이션 분석

Spark UI는 어플리케이션의 성능 병목 현상을 진단하고 내부 동작을 이해할 수 있는 강력한 웹 도구입니다.

### 주요 탭별 기능

- **Jobs & Stages**: 어플리케이션이 어떤 작업(Job)과 단계(Stage)로 나뉘었는지 확인. 실행 시간이 유독 긴 단계가 있다면 개별 태스크(Task) 단위로 추적 가능.
- **Executors**: 각 실행기가 사용하는 리소스(코어, 메모리, 디스크) 현황 파악. 가비지 컬렉션(GC) 시간이 길다면 메모리 부족을 의심해야 함.
- **Storage**: `cache()`나 `persist()`를 통해 메모리에 적재된 데이터프레임의 상세 정보(파티션 수, 메모리 사용량 등) 확인.
- **SQL**: 실행된 SQL 쿼리의 물리적 실행 계획(Physical Plan) 시각화. 데이터 스캔 량, 셔플 발생 여부 등을 지표로 확인 가능.
- **Environment**: JVM 버전, 라이브러리(Jar) 경로, 환경 변수 등 현재 클러스터의 설정값 확인.

## 3. 성능 진단의 실마리 (Breadcrumbs)

UI를 통해 다음과 같은 이상 징후를 포착하고 해결할 수 있습니다.

- **데이터 스큐(Data Skew)**: 특정 태스크의 실행 시간이 평균보다 훨씬 길다면 데이터가 특정 파티션에 쏠려 있는 상태.
- **I/O 병목**: 셔플 읽기 차단 시간(Shuffle Read Blocked Time)이 길다면 네트워크나 디스크 성능 문제 가능성.
- **메모리 부족**: 높은 GC 시간은 힙 메모리에 객체가 너무 많음을 의미
