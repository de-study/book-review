## 1. 최적의 스토리지 솔루션의 중요성

데이터 파이프라인의 신뢰성은 **스토리지 계층의 품질**에 직결됨.

```
데이터 파이프라인에서 스토리지 계층의 역할
--------------------------------------------------------------
수집 → 변환 → [스토리지] → 분석/서빙
               ▲
               │
  여기서 발생하는 문제가 하위 전체에 영향
  - 데이터 손상      → 잘못된 분석 결과
  - 스키마 불일치    → 파이프라인 중단
  - 부분 쓰기 실패   → 불완전한 데이터 노출
--------------------------------------------------------------
```

데이터 볼륨과 다양성이 증가할수록, 스토리지 계층에서 다음 세 가지를 동시에 만족해야 함:

| 요구사항 | 설명 |
|---------|------|
| **확장성** | 스토리지 솔루션은 데이터 볼륨에 맞게 확장할 수 있어야 하며 워크로드에 필요한 읽기/쓰기 처리량 및 지연 시간 제공해야 함 |
| **트랜잭션 지원** | ACID 트랜잭션에 대한 지원은 최종 결과의 품질 보장에 필수적 |
| **다양한 데이터 형식 지원** | 스토리지 솔루션은 비정형, 반정형 및 정형 데이터 저장할 수 있어야 함 |
| **다양한 워크로드 지원** | SQL 워크로드, 전통적 ETL 같은 배치 워크로드, 실시간 모니터링 및 장애 알람 같은 스트리밍 워크로드, ML 및 AI 워크로드 |
| **개방성** | 광범위한 워크로드 지원 위해 오픈 데이터 형식으로 데이터 저장 및 표준 API를 사용하여 다양한 도구와 데이터에 액세스 가능 |

## 2. 데이터베이스

### 2-1. 데이터베이스 개요

데이터베이스는 구조화된 데이터를 SQL 쿼리 사용해서 읽을 수 있는 테이블로 저장하도록 설계
데이터는 데이터베이스 관리 시스템이 데이터 저장 및 처리를 모두 최적화할 수 있게 하는 엄격한 스키마 준수해야 한다.

- **목적**
  - 정형 데이터의 안정적인 저장 및 트랜잭션 기반 조회
  - 결제, 사용자 정보처럼 정합성이 최우선인 데이터 관리에 특화

- **이유**
  - ACID 트랜잭션으로 데이터 정합성을 엔진 수준에서 보장
  - 여러 사용자가 동시에 읽고 쓰더라도 데이터 손상 없이 일관성 유지 가능

- **결과**
  - 금융 거래, 재고 관리 등 정합성이 중요한 시스템에서 안정적으로 운영 가능
  - 인덱스 기반 포인트 쿼리(`WHERE id = 123`)는 밀리초 단위 응답 가능

- **장점**
  - ACID 트랜잭션으로 데이터 손상 방지
  - 인덱스를 통한 빠른 포인트 조회
  - 수십 년 이상 성숙한 운영 도구 생태계 (모니터링, 백업, 복구)

- **단점**
  - 비정형 데이터(JSON, 이미지, 로그) 저장 불가 또는 매우 비효율적
  - 수평 확장(Scale-out) 구조가 아니어서 대용량 데이터 적재 시 비용 급증
  - 테라바이트 이상 풀스캔 분석 쿼리에 부적합

| SQL 워크로드 종류 | 설명 |
|---------|------|
| **온라인 트랜잭션 처리 (OLTP) 워크로드** | 일반적으로 한 번에 몇 개의 레코드 읽거나 업데이트 하는 간단한 쿼리. 높은 동시성, 짧은 지연 시간이 특징 |
| **온라인 분석 처리 (OLAP)** | 일반적으로 많은 레코드에 대한 높은 처리량 스캔이 필요한 복잡한 쿼리 (집계 및 조인 포함) |

아파치 스파크는 주로 OLTP 워크로드가 아닌 **OLAP 워크로드용으로 설계**된 쿼리 엔진이다


```
ACID 트랜잭션 보장 구조
--------------------------------------------------------------
A (Atomicity)   — 전체 성공 or 전체 실패. 절반만 반영되는 상황 없음
C (Consistency) — 트랜잭션 전후 데이터가 항상 유효한 상태 유지
I (Isolation)   — 동시 실행 중인 트랜잭션이 서로 간섭하지 않음
D (Durability)  — 커밋된 데이터는 장애 후에도 반드시 보존

예: 계좌 이체
  BEGIN
    계좌 A -10만원   ← 이 두 연산이
    계좌 B +10만원   ← 반드시 같이 성공해야 함
  COMMIT

  중간에 장애 발생 시 → 전체 롤백, 이전 상태로 복원
--------------------------------------------------------------
```

### 2-2. 아파치 스파크로 데이터베이스 읽기/쓰기

Spark는 JDBC 커넥터를 통해 대부분의 RDBMS와 연결 가능.  
대용량 읽기 시 `partitionColumn` 옵션으로 병렬 처리를 활성화하는 것이 핵심.

```python
# PostgreSQL에서 읽기
jdbc_url = "jdbc:postgresql://host:5432/mydb"
properties = {"user": "user", "password": "pw", "driver": "org.postgresql.Driver"}

df = (
    spark.read.jdbc(
        url=jdbc_url,
        table="orders",
        column="order_id",       # 파티셔닝 기준 컬럼 (숫자형)
        lowerBound=1,
        upperBound=1000000,
        numPartitions=10,        # 병렬 읽기 파티션 수
        properties=properties,
    )
)

# PostgreSQL에 쓰기
df.write.jdbc(
    url=jdbc_url,
    table="orders_summary",
    mode="append",
    properties=properties,
)
```

> **트러블 로그** — `numPartitions` 없이 대용량 테이블을 읽으면 Spark가 단일 태스크로 전체를 읽어 드라이버 OOM 발생.  
> 예: 5천만 건 orders 테이블을 `numPartitions` 없이 읽으면 드라이버 메모리 8GB가 통째로 채워져 파이프라인이 죽음.  
> `partitionColumn`은 `order_id` 같은 균등 분포된 정수형 PK를 사용할 것. `created_at` 같은 datetime은 특정 날짜에 데이터가 몰릴 경우 파티션 skew 발생.

### 2-3. 데이터베이스의 한계

```
데이터베이스로 감당하기 어려운 시나리오
--------------------------------------------------------------
[10년간 분석 워크로드 2가지 새로운 추세]

① 테라바이트~페타바이트 규모의 분석 쿼리
   → 빅데이터 도래와 함께 트렌드와 사용자 행동 이해하기 위해 모든 것 (e.g. 페이지뷰, 클릭 등) 측정하고 수집
   → 풀스캔 시 수십 분 소요, 운영 DB에 부하 집중
   
② 분석의 다양성 증가
   → 더 깊은 인사이트 필요해져서 머신러닝 및 딥러닝 처럼 복잡한 분석 폭발적으로 성장

[데이터베이스의 제약 사항]

① 수직 확장(Scale-up)만 가능 (데이터베이스 확장하는데 많은 비용 필요)
   → 데이터 폭증 시 고사양 서버 교체 외에 선택지가 없음
   → 처리 엔진을 위해 데이터를 병렬로 처리하는 수평 확장이 필요하지만 오픈소스 데이터베이스는 분산 처리 수행 위한 확장 가능하게 설계되지 않음

② 데이터베이스는 비SQL 기반 분석 잘 지원하지 않음
   → 일반적으로 해당 데이터베이스의 SQL 처리 엔진만 읽을 수 있게 복잡한 형식으로 데이터 저장
   → 머신러닝 및 딥러닝 같은 다른 처리 도구가 데이터에 효율적으로 액세스 못한다

③ 스키마 변경(컬럼 추가/삭제) 비용이 높음
   → 대형 테이블 ALTER TABLE은 테이블 잠금 발생 가능

④ 비정형 로그, JSON 이벤트, 이미지 등 다양한 포맷 저장 불가
--------------------------------------------------------------
```

## 3. 데이터 레이크

### 3-1. 데이터 레이크 개요

데이터 레이크는 범용 하드웨어에서 실행되고, 수평으로 쉽게 확장되는 분산 스토리지 솔루션
- 데이터 레이크 아키텍처는 데이터베이스와 달리 분산 컴퓨팅 시스템에서 분산 스토리지 시스템을 분리
- 이를 통해 각 시스템은 워크로드에 의한 필요에 따라 확장 가능

데이터는 모든 처리 엔진이 표준 API 사용하여 읽고 쓸 수 있게 오픈 파일 형식으로 저장


- **목적**
  - 다양한 포맷의 대용량 데이터를 저비용 오브젝트 스토리지에 원시 상태로 보관
  - 분석 전까지 가공하지 않고 일단 모아두는 "원시 데이터 저장소" 역할

- **이유**
  - S3, GCS, ADLS 같은 오브젝트 스토리지는 RDBMS 대비 약 1/10 수준의 비용
  - 스키마를 미리 정의하지 않아도 되므로 정형/반정형/비정형 데이터를 모두 수용 가능

- **결과**
  - 로그, JSON 이벤트, Parquet, 이미지 등을 한 곳에 보관하고
  - Spark, Presto, Athena 등 다양한 쿼리 엔진으로 필요할 때 처리 가능

- **장점**
  - 낮은 스토리지 비용 (S3 Standard 기준 ~$0.023/GB/월)
  - 데이터 크기에 관계없이 수평 확장 가능
  - Parquet, ORC, JSON, CSV, Avro 등 다양한 포맷 지원
  - 현재의 워크로드에 가장 적합한 스토리지 시스템, 오픈 데이터 형식 및 처리 엔진을 선택할 수 있는 유연성이 데이터베이스 대비 데이터 레이크의 큰 장점

- **단점**
  - ACID 트랜잭션 없음 → 쓰기 중 실패 시 부분 파일이 노출될 수 있음
  - 스키마 관리 부재 → 업스트림 컬럼 변경을 자동으로 감지하지 못함
  - 데이터 품질 보장 어려움 → 잘못된 데이터가 그대로 쌓이는 "데이터 늪" 위험
  - 소파일 문제 → 스트리밍 파이프라인 운영 시 파일 수 폭증으로 점진적 성능 저하

```
데이터 레이크 구조 예시
--------------------------------------------------------------
s3://my-data-lake/
├── raw/
│   ├── events/year=2024/month=01/day=15/*.json   ← 원시 이벤트
│   ├── logs/year=2024/month=01/day=15/*.gz       ← 압축 로그
│   └── images/*.jpg                              ← 비정형 데이터
├── processed/
│   └── orders/year=2024/month=01/*.parquet       ← 변환된 정형 데이터
└── serving/
    └── daily_summary/*.parquet                   ← 분석용 집계 데이터

Hive 메타스토어 또는 AWS Glue Catalog로 테이블 메타데이터 관리
--------------------------------------------------------------
```

### 3-2. 아파치 스파크로 데이터 레이크 읽기/쓰기

아파치 스파크는 자체 데이터 레이크를 구축할 때 사용할 수 있는 최고의 처리 엔진 중 하나이다.
- 다양한 워크로드 지원 (배치 처리, ETL, SQL, Structured Streaming 통한 스트림 처리, MLlib 통한 머신러닝 등)
- 다양한 파일 형식 지원 (비정형, 반정형, 정형)
- 다양한 파일 시스템 지원 (하둡 파일 시스템 API를 지원하는 모든 스토리지 시스템에 대해 데이터 접근 가능)

AWS S3와 같은 클라우드 스토리지 기반 파일 시스템의 경우 안전한 방식으로 파일 시스템에 액세스할 수 있도록 스파크를 구성해야 한다.

```python
# Parquet으로 파티셔닝 저장 (일반적인 데이터 레이크 패턴)
df.write \
    .partitionBy("year", "month", "day") \
    .mode("append") \
    .parquet("s3://my-data-lake/processed/orders/")

# 파티션 필터 푸시다운으로 효율적 읽기
orders = spark.read.parquet("s3://my-data-lake/processed/orders/") \
    .filter("year = '2024' AND month = '01'")   # 해당 파티션만 스캔

# Hive 메타스토어 등록 후 SQL로 조회
spark.sql("""
    CREATE TABLE IF NOT EXISTS orders
    USING parquet
    LOCATION 's3://my-data-lake/processed/orders/'
""")
```

> **트러블 로그** — 파티션 컬럼을 잘못 고르면 오히려 성능이 크게 나빠짐.  
> 예: `user_id`로 파티셔닝하면 MAU 100만 기준 하루에만 수백만 개의 소파일 생성.  
> S3에서 `LIST` 호출 한 번에 최대 1,000개 객체만 반환하므로, 파일 수가 많아질수록 메타데이터 조회만으로 수십 초 소요.  
> 일반적으로 `year/month/day` 계층 구조가 가장 무난하며, 쿼리 패턴에서 가장 자주 필터링하는 컬럼을 기준으로 선택할 것.

### 3-3. 데이터 레이크의 한계

```
데이터 레이크의 주요 문제점
--------------------------------------------------------------
① ACID 미지원
   - 원자성과 독립성
     - 분산 방식으로 파일을 데이터 레이크에 기록하던 중 작업이 실패하면 작성된 파일을 롤백하는 매커니즘 없음
     - 쓰기 도중 실패 시 부분 파일이 그대로 노출
     - 동시 쓰기 시 데이터 충돌 및 손상 위험 (동시성 제어 불가)
   - 일관성
     - 실패한 쓰기에 대한 원자성 부족으로 인해 데이터에 대한 일관성 잃게 됨 (데이터 정합성 결여)
   
② 스키마 관리 부재 ("스키마 온 리드")
   - 업스트림이 컬럼을 추가/변경해도 자동 감지 불가
   - 파이프라인 실행 시점에야 스키마 오류 발견

③ 데이터 수정의 어려움
   - Parquet 파일은 불변(immutable) → 특정 행 수정 불가
   - GDPR 삭제 요청, 잘못된 데이터 수정 시 전체 재처리 필요

④ 소파일 문제
   - 스트리밍 파이프라인이 수십 초마다 파일 생성
   - S3 LIST 호출 비용 증가, 쿼리 성능 저하

--------------------------------------------------------------
→ 이 한계를 극복한 것이 레이크하우스
--------------------------------------------------------------
```

## 4. 레이크하우스: 스토리지 솔루션 진화의 다음 단계

레이크하우스 = **데이터 레이크의 저비용·확장성** + **데이터베이스의 신뢰성(ACID, 스키마 관리)**

```
레이크하우스가 해결하는 문제
------------------------------------------------------------------------------
데이터 레이크 문제               레이크하우스 해결책
--------------------------  --------------------------------------------------
ACID 미지원                   →  트랜잭션 로그 기반 ACID 보장
스키마 관리 부재                →  스키마 강제(Schema Enforcement)
데이터 수정 불가                →  Upsert / Delete 지원
소파일 문제                    →  파일 컴팩션 자동화
이력 추적 불가                  →  버전 관리 + 타임 트래블

                            → 오픈 포맷의 다양한 데이터 유형 지원
                            → 다양한 워크로드 지원
                            → 데이터 무결성에 대해 추론하고 데이터 변경 사항 감사 가능
-----------------------------------------------------------------------------
```

레이크하우스 구축 위한 오픈 소스 시스템 (아파치 후디, 아파치 아이스버그, 델타 레이크)
아래와 같은 오픈 데이터 스토리지 형식으로 구성
- 확장 가능한 파일 시스템에 정형 파일 형식으로 대용량 데이터 저장
- 트랜잭션 로그 유지하여 데이터에 대한 원자적 변경 타임라인 기록 (데이터베이스와 유사)
- 로그를 사용하여 테이블 데이터의 버전 정의하고 읽기와 쓰기 사이 스냅샷 격리 보장 제공
- 아파치 스파크 사용하여 테이블에 대한 읽기 및 쓰기 지원

### 4-1. 아파치 후디 (Apache Hudi)

- **목적**
  - 스트리밍 Upsert와 증분 처리에 특화된 레이크하우스 포맷
  - DB 변경사항을 실시간으로 레이크에 반영하는 CDC (Change Data Capture) 파이프라인에 최적화

- **이유**
  - MySQL → Kafka → 레이크로 이어지는 CDC 파이프라인에서
  - 레코드 단위 Upsert를 오브젝트 스토리지 위에서 효율적으로 수행해야 하는 수요 대응

- **결과**
  - 레코드 단위 Upsert 지원으로 "마지막 커밋 이후 변경분만 읽기" 패턴 구현 가능
  - 전체 재처리 없이 변경된 레코드만 처리하여 파이프라인 비용 절감

- **장점**
  - Kafka → Hudi 스트리밍 Upsert 파이프라인 구성이 다른 포맷보다 용이
  - 증분 읽기(Incremental Query)로 처리 비용 절감
  - MOR 테이블 타입으로 실시간 Upsert와 배치 분석 사이의 트레이드오프 조절 가능
  - 빠르고 플러그 가능한 인덱싱으로 Upsert
  - 롤백 지원 통한 데이터 원자성 게시
  - 테이블에 대한 증분 변경 읽기
  - 데이터 복구를 위한 저장점
  - 통계를 이용한 파일 크기 및 레이아웃 관리
  - 행 및 열 데이터의 비동기 압축

- **단점**
  - 설정 옵션이 많고 학습 곡선이 가파름
  - Delta Lake 대비 Spark 통합 성숙도가 낮음
  - 운영 중 트러블슈팅 난이도가 높음

| 테이블 타입 | 특징 | 적합한 용도 |
|-----------|------|-----------|
| Copy-On-Write (COW) | 쓰기 시 파일 재작성, 읽기 최적화 | 배치 분석 중심 |
| Merge-On-Read (MOR) | 쓰기 시 델타 로그 추가, 읽기 시 머지 | 실시간 Upsert 중심 |

### 4-2. 아파치 아이스버그 (Apache Iceberg)

- **목적**
  - 아이스버그는 단일 테이블에서 페타바이트까지 증가할 수 있는 확장성과 스키마 진화 특성을 갖는 범용 데이터 스토리지에 더 중점
    - 후디는 키/값 데이터를 Upsert하는 데 중점을 둔다는 점에서 다르다
  - 페타바이트 규모 테이블을 Spark, Flink, Trino, Hive 등 다중 엔진으로 안전하게 접근
  - 하나의 테이블을 여러 팀이 서로 다른 엔진으로 읽고 쓰는 환경에 최적화

- **이유**
  - Spark로 쓰고 Trino로 읽는 멀티 엔진 환경에서 메타데이터 충돌이 발생하기 쉬움
  - Iceberg는 Optimistic Concurrency Control로 이 문제를 해결

- **결과**
  - 파티션 진화(기존 데이터 재작성 없이 파티션 전략 변경) 지원
  - 숨겨진 파티셔닝(테이블 행에 대한 파티션값 내부적으로 생성, 쿼리 작성 시 파티션 컬럼 명시 불필요)으로 쿼리 편의성 향상

- **장점**
  - 멀티 엔진 지원 측면에서 세 포맷 중 가장 성숙
  - 대규모 테이블 메타데이터 관리 효율적 (파일 단위 통계 저장)
  - 파티션 진화 지원으로 파티션 전략 변경 시 재처리 불필요
  - 스키마 진화 가능
  - ID 또는 타임스탬프별로 특정 테이블 스냅샷 쿼리할 수 있는 시간 탐색 (Time Travel) 기능
  - 오류 수정 위한 이전 버전으로 롤백 기능
  - 여러 동시 쓰기 작업 간에도 직렬화 가능한 격리 (Serializable Isolation) 기능

- **단점**
  - Delta Lake 대비 Spark 네이티브 기능(Z-Ordering 등) 부족
  - 생태계 성숙도가 Delta Lake보다 늦게 시작되어 레퍼런스 자료가 상대적으로 적음

### 4-3. 델타 레이크 (Delta Lake)

- **목적**
  - Spark 기반 파이프라인에서 신뢰할 수 있는 레이크하우스 구축
  - 배치와 스트리밍을 동일한 테이블과 API로 처리하는 통합 아키텍처 구현

- **이유**
  - Databricks가 Spark와의 긴밀한 통합을 목표로 개발
  - Spark의 모든 기능(Structured Streaming, ML, SQL)과 자연스럽게 연동

- **결과**
  - ACID 트랜잭션, 스키마 강제/진화, 타임 트래블, MERGE/UPDATE/DELETE,
  - Structured Streaming 통합을 하나의 포맷에서 모두 제공
    - Structured Streaming Source 및 Sink 사용하여 테이블에서 스트리밍 읽기 및 쓰기

- **장점**
  - Spark 통합 측면에서 세 포맷 중 가장 성숙하고 안정적
  - 운영 편의성 높음 (VACUUM, OPTIMIZE 등 관리 명령 내장)
  - 오픈소스 + Databricks 상용 지원 모두 가능
  - 테이블 스키마 명시적으로 변경하거나 데이터 프레임의 쓰기 중에도 DF 스키마를 테이블 스키마에 암시적으로 병합하여 스키마 변경 가능
  - ID 또는 타임스탬프별로 특정 테이블 스냅샷 쿼리할 수 있는 Time Travel 기능
  - 오류 수정 위한 이전 버전으로 롤백 기능
  - SQL, 배치 처리 또는 스트리밍 작업 수행하는 여러 동시 작성기 간의 직렬화 가능한 쿼리

- **단점**
  - 멀티 엔진 지원은 Iceberg 대비 부족 (개선 진행 중)
  - Delta 포맷에 대한 종속성 존재

| 기능 | 후디 | 아이스버그 | 델타 레이크 |
|------|------|----------|-----------|
| ACID 트랜잭션 | ✅ | ✅ | ✅ |
| 스트리밍 Upsert | ✅ (강점) | ✅ | ✅ |
| 타임 트래블 | ✅ | ✅ | ✅ |
| 멀티 엔진 지원 | 보통 | **최강** | 보통 |
| Spark 통합 | 보통 | 보통 | **최강** |
| 파티션 진화 | 제한적 | ✅ (강점) | ✅ |

## 5. 아파치 스파크 및 델타 레이크로 레이크하우스 구축

### 5-1. 델타 레이크와 아파치 스파크 구성

```python
# SparkSession에 Delta Lake 패키지 추가
spark = (
    SparkSession.builder
    .appName("DeltaLakeDemo")
    .config("spark.jars.packages", "io.delta:delta-core_2.12:2.4.0")
    .config("spark.sql.extensions", "io.delta.sql.DeltaSparkSessionExtension")
    .config(
        "spark.sql.catalog.spark_catalog",
        "org.apache.spark.sql.delta.catalog.DeltaCatalog",
    )
    .getOrCreate()
)
```

### 5-2. 델타 레이크 테이블에 데이터 로드 (배치)

```python
# Delta 형식으로 최초 저장
df.write \
    .format("delta") \
    .mode("overwrite") \
    .save("/delta/loans/")

# 이후 배치 append
new_loans.write \
    .format("delta") \
    .mode("append") \
    .save("/delta/loans/")

# SQL로 테이블 등록 후 조회
spark.sql("CREATE TABLE loans USING delta LOCATION '/delta/loans/'")
spark.sql("SELECT * FROM loans WHERE loan_status = 'active'").show()
```

> **트러블 로그** — `overwrite` 모드는 기존 데이터를 전부 덮어씀.  
> Delta Lake는 이를 트랜잭션으로 처리하므로 쓰기 중 쿼리가 들어와도 이전 스냅샷을 읽음.  
> 하지만 운영 중인 테이블에 실수로 `overwrite`를 날린 적이 있다면, 이전 데이터로 되돌리려면 타임 트래블이 필요.  
> 습관적으로 `append`를 기본값으로 쓰고, `overwrite`는 의도적인 경우에만 명시할 것.

### 5-3. 델타 레이크 테이블에 데이터 스트림 로드

배치와 스트리밍을 동일한 Delta 테이블에 혼합 쓰기 가능.  
이것이 Delta Lake의 핵심 가치 중 하나 — **배치/스트리밍 통합 테이블**.

```
델타 레이크의 가장 혁신적인 점은 데이터의 유입 속도(Velocity)나 처리 방식(Batch vs Streaming)에 상관없이 
하나의 테이블에서 모든 데이터를 관리할 수 있다는 것

1. 기존 아키텍처의 한계: 람다 아키텍처 (Lambda Architecture)
과거에는 실시간 데이터의 신선도와 과거 데이터의 정확성을 모두 확보하기 위해 두 가지 경로를 별도로 운영
- Batch Layer: 대규모 과거 데이터를 처리하며 정확도가 높지만 느림.
- Speed Layer: 실시간 이벤트를 즉시 처리하지만 데이터 유실 가능성이 있고 정합성 유지가 어려움.
- 문제점: 동일한 비즈니스 로직을 두 번 구현해야 하며, 두 레이어의 데이터를 합쳐서 조회할 때 정합성을 맞추는 작업이 매우 복잡함.

2. Delta Lake를 통한 통합 (Unified Table)
델타 레이크는 ACID 트랜잭션 기능을 데이터 레이크에 도입함으로써, 배치와 스트리밍 작업이 동일한 테이블에 동시에 접근하더라도 데이터 무결성을 보장.

① 단일 API 및 저장소
데이터 엔지니어는 소스 시스템이 무엇이든 관계없이 동일한 Delta Table 형식으로 데이터를 적재 가능.
- 배치 처리: 대량의 이력 데이터를 한 번에 Append하거나 Overwrite.
- 스트리밍 처리: Kafka 등에서 유입되는 실시간 데이터를 초 단위로 Append.

② 낙관적 동시성 제어 (Optimistic Concurrency Control)
- 스트리밍 프로세스가 데이터를 쓰고 있는 도중에도, 배치 프로세스가 동일한 테이블에 과거 데이터를 보정(Backfill)하거나 추가하는 작업이 가능.
- 트랜잭션 로그(Delta Log)가 각 작업의 순서와 성공 여부를 관리하여 데이터 손상을 방지.

3. 실무적 이점 (Business & Engineering Value)

① 단일 진실 공급원 (Single Source of Truth)
분석가는 실시간 테이블과 배치 테이블을 따로 Join할 필요가 없다. 
단 하나의 델타 테이블만 조회하면 과거부터 현재까지의 전체 데이터를 일관된 상태로 얻을 수 있다.

② 효율적인 데이터 보충 (Backfill)
실시간 파이프라인에서 데이터 누락이 발생하더라도, 스트리밍 서비스를 중단할 필요 없이 배치 작업을 통해 해당 구간의 데이터만 정교하게 다시 밀어 넣을 수 있다.

③ 아키텍처 단순화 (Kappa Architecture)
복잡한 람다 아키텍처에서 벗어나, 모든 데이터를 스트림으로 간주하고 처리하는 카파 아키텍처(Kappa Architecture) 혹은 레이크하우스(Lakehouse) 아키텍처를 쉽게 구현할 수 있다.
```

```python
# Kafka 스트림 → Delta 테이블로 지속 쓰기
stream_query = (
    spark.readStream
    .format("kafka")
    .option("kafka.bootstrap.servers", "broker:9092")
    .option("subscribe", "loan-events")
    .load()
    .selectExpr("CAST(value AS STRING)")
    .select(from_json(col("value"), loan_schema).alias("data"))
    .select("data.*")
    .writeStream
    .format("delta")                                          # Delta로 쓰기
    .outputMode("append")
    .option("checkpointLocation", "/delta/checkpoints/loans") # 체크포인트 필수
    .start("/delta/loans/")
)
```

```
                                                     ├──► Delta 테이블
[스트리밍]   ──► Structured Streaming ────────────────►┘
                                                            │
                                                       트랜잭션 로그
                                                       (_delta_log/)
                                                            │
                                           ┌────────────────┴────────────────┐
                                           ▼                                 ▼
                                     배치 분석 쿼리                         스트리밍 읽기
                                 (Spark SQL, Presto 등)         (readStream.format("delta"))
------------------------------------------------------------------------------------------------------------------
```

### 5-4. 데이터 손상을 방지하기 위해 쓰기 시 스키마 적용 (Schema Enforcement)

- **목적**
  - 테이블 스키마와 맞지 않는 데이터가 저장되는 것을 엔진 수준에서 차단
  - 업스트림 변경이 다운스트림 테이블을 오염시키는 상황 방지

- **이유**
  - 데이터 레이크에서는 업스트림이 컬럼을 삭제하거나 타입을 바꿔도 그대로 저장됨
  - Delta는 쓰기 시점에 스키마를 검증해 이런 오염을 사전 차단

- **결과**
  - 스키마 불일치 시 `AnalysisException` 발생 및 쓰기 실패
  - 기존 데이터는 오염되지 않으며, 문제를 쓰기 시점에 즉시 인지 가능

- **장점**
  - 별도 데이터 품질 검사 없이도 스키마 계약(contract) 자동 강제
  - 잘못된 데이터가 싱크까지 흘러가지 않아 디버깅 비용 절감
  - 팀 간 스키마 계약의 기술적 보장 수단으로 활용 가능

- **단점**
  - 의도한 스키마 변경(컬럼 추가 등)도 일단 거부됨
  - 스키마 진화 옵션(`mergeSchema`)을 명시적으로 활성화해야 적용 가능

```python
# 스키마 불일치 예시
# 기존 Delta 테이블 스키마: loan_id (int), amount (int), status (string)

# 새 데이터에 기존 없는 컬럼 'risk_score' 추가
new_df = spark.createDataFrame(
    [(1001, 50000, "active", 0.87)],
    ["loan_id", "amount", "status", "risk_score"],  # 새 컬럼 포함
)

# → 기본 설정에서 아래와 같이 실패
new_df.write.format("delta").mode("append").save("/delta/loans/")
# AnalysisException: A schema mismatch detected when writing to the Delta table.
```

> **트러블 로그** — 스키마 강제는 데이터 품질의 첫 번째 방어선.  
> 운영 중 파이프라인에서 갑자기 `AnalysisException`이 뜨면 십중팔구 업스트림 스키마 변경이 원인.  
> 예: 광고팀이 클릭 이벤트 스키마에 `campaign_id` 컬럼을 추가했는데 알림 없이 배포 → 하위 파이프라인 전체 중단.  
> 이런 상황이 반복되지 않도록 업스트림 팀과 **스키마 변경 알림 프로세스**를 사전에 합의해 둘 것.

### 5-5. 변화하는 데이터를 수용하기 위해 진화하는 스키마 (Schema Evolution)

- **목적**
  - 테이블을 재생성하거나 전체 재처리 없이 새 컬럼을 안전하게 추가
  - 비즈니스 요구 변화에 따라 스키마가 점진적으로 확장되는 상황 대응

- **이유**
  - 컬럼 추가는 불가피한데, 매번 테이블 재작성은 운영 비용이 너무 큼
  - `mergeSchema` 옵션 하나로 신규 컬럼을 안전하게 흡수 가능

- **결과**
  - 신규 컬럼이 Delta 테이블 스키마에 자동으로 추가됨
  - 기존 행의 해당 컬럼값은 `null`로 채워지며, 이전 데이터는 영향 없음

- **장점**
  - 파이프라인 중단 없이 점진적 스키마 확장 가능
  - 기존 행은 `null`로 자동 처리되어 별도 백필 로직 불필요

- **단점**
  - 컬럼 타입 변경(예: `int` → `string`)은 지원하지 않음
  - 컬럼 삭제도 직접 지원 안 됨 → 삭제 시 테이블 재작성 필요

```python
# 스키마 진화 활성화: 신규 컬럼 자동 추가
new_df.write \
    .format("delta") \
    .mode("append") \
    .option("mergeSchema", "true") \    # 이 옵션 하나로 스키마 진화 허용
    .save("/delta/loans/")

# SparkSession 전역 설정으로도 가능
spark.conf.set("spark.databricks.delta.schema.autoMerge.enabled", "true")
```

```
스키마 진화 전후 비교
--------------------------------------------------------------
기존 스키마:
  loan_id (int) | amount (int) | status (string)

신규 데이터 (새 컬럼 포함):
  loan_id (int) | amount (int) | status (string) | risk_score (double)
                                                       ↑ 신규 컬럼

mergeSchema=true 적용 후 스키마:
  loan_id (int) | amount (int) | status (string) | risk_score (double)

  → 기존 행의 risk_score = null 자동 처리
--------------------------------------------------------------
```

### 5-6. 기존 데이터 변환 (Update / Delete / Merge)

Delta Lake는 불변 파일 위에 트랜잭션 로그를 얹어 **행 단위 수정**을 지원.  
내부적으로는 영향받는 Parquet 파일을 재작성하는 방식으로 동작.

```python
from delta.tables import DeltaTable

delta_table = DeltaTable.forPath(spark, "/delta/loans/")

# UPDATE — 특정 조건의 행 값 변경
delta_table.update(
    condition="loan_status = 'active' AND due_date < current_date()",
    set={"loan_status": "'overdue'"},
)

# DELETE — GDPR 삭제 요청 처리
delta_table.delete("borrower_id = 'user_12345'")

# MERGE (Upsert) — CDC 패턴에서 자주 사용
(
    delta_table.alias("target")
    .merge(
        new_loans.alias("source"),
        "target.loan_id = source.loan_id",
    )
    .whenMatchedUpdate(set={"loan_status": "source.loan_status", "amount": "source.amount"})
    .whenNotMatchedInsertAll()
    .execute()
)
```

> **트러블 로그** — `MERGE`는 강력하지만 대규모 테이블에서 성능 저하가 심할 수 있음.  
> 예: 1억 건 테이블에 1만 건 MERGE를 날렸는데 전체 스캔이 발생해 30분이 걸린 사례도 있음.  
> 조인 키(`loan_id`)가 파티션 컬럼과 일치하는지 먼저 확인하고,  
> `spark.databricks.delta.merge.repartitionBeforeWrite.enabled`를 활성화해 쓰기 skew를 방지할 것.

### 5-7. 작업 내역으로 데이터 변경 검사 (History & Audit)

- **목적**
  - 테이블에 어떤 작업이, 언제, 어떤 파라미터로 수행됐는지 감사 추적
  - 데이터 품질 이슈 발생 시 원인 시점을 빠르게 파악하는 수단

- **이유**
  - "어제 오전과 오늘 집계 결과가 왜 다른가?" 같은 질문에 답하려면
  - 테이블에 어떤 작업이 있었는지 추적 가능해야 함
  - 컴플라이언스 요구사항(감사 로그 보관 의무)에도 대응 가능

- **결과**
  - `DESCRIBE HISTORY` 또는 `DeltaTable.history()`로 버전별 작업 이력 조회
  - 버전 번호를 기준으로 특정 시점으로 롤백 가능

- **장점**
  - 별도 감사 로그 시스템 구축 없이 Delta 자체에서 이력 관리
  - 특정 버전으로 롤백 가능 (타임 트래블과 결합)
  - `operationParameters` 필드에서 실행된 쿼리 조건까지 기록됨

- **단점**
  - 이력은 기본 30일 보관 후 `VACUUM` 실행 시 삭제됨
  - 장기 보관이 필요하면 별도 아카이브 파이프라인 구성 필요

```python
# 테이블 변경 이력 조회
delta_table = DeltaTable.forPath(spark, "/delta/loans/")
history_df = delta_table.history()
history_df.select("version", "timestamp", "operation", "operationParameters").show()

# SQL로도 가능
spark.sql("DESCRIBE HISTORY delta.`/delta/loans/`").show()
```

### 5-8. 시간 탐색을 사용하여 테이블의 이전 스냅샷 쿼리 (Time Travel)

- **목적**
  - 과거 특정 시점 또는 버전의 데이터를 현재 테이블 변경 없이 조회
  - 실수로 삭제된 데이터 복구, 과거 시점 비교, ML 실험 재현 등에 활용

- **이유**
  - Delta는 쓰기 시 기존 파일을 삭제하지 않고 새 파일을 추가 + 트랜잭션 로그 갱신
  - 이 구조 덕분에 과거 버전의 파일 집합을 그대로 참조 가능

- **결과**
  - 버전 번호 또는 타임스탬프 기준으로 과거 스냅샷을 일반 DataFrame처럼 읽기 가능
  - 읽기 전용 조회이므로 현재 테이블 상태에 영향 없음

- **장점**
  - 실수로 DELETE/OVERWRITE한 데이터를 별도 백업 없이 복구 가능
  - ML 학습 데이터셋을 특정 버전으로 고정하여 실험 재현성 확보
  - 추가 스토리지 비용 최소화 (변경분만 저장하는 구조)

- **단점**
  - `VACUUM` 실행 후에는 오래된 버전 파일이 삭제되어 타임 트래블 범위 축소
  - 기본 보관 기간이 7일로 짧아 운영 전 설정 조정 필요

```python
# 버전 기준 과거 데이터 조회
loans_v3 = spark.read \
    .format("delta") \
    .option("versionAsOf", 3) \
    .load("/delta/loans/")

# 타임스탬프 기준 과거 데이터 조회
loans_yesterday = spark.read \
    .format("delta") \
    .option("timestampAsOf", "2024-01-14 00:00:00") \
    .load("/delta/loans/")

# SQL로도 가능
spark.sql("""
    SELECT * FROM loans
    TIMESTAMP AS OF '2024-01-14 00:00:00'
    WHERE loan_status = 'active'
""")

# 실수로 DELETE한 데이터 복구 패턴
loans_before_delete = spark.read \
    .format("delta") \
    .option("versionAsOf", 3) \      # DELETE 이전 버전
    .load("/delta/loans/")

loans_before_delete.write \
    .format("delta") \
    .mode("overwrite") \
    .option("overwriteSchema", "true") \
    .save("/delta/loans/")            # 복구 완료
```

> **트러블 로그** — `VACUUM`은 기본적으로 7일 이전 파일을 삭제함.  
> 타임 트래블 범위가 줄어들기 때문에, 처음 테이블을 만들 때부터 보관 기간을 명시적으로 설정해 두는 것이 좋음.  
> 예: 감사 목적으로 30일치 이력이 필요하다면 `ALTER TABLE loans SET TBLPROPERTIES ('delta.logRetentionDuration' = 'interval 30 days')`.  
> 실수로 `VACUUM 0 HOURS`를 실행하면 진행 중인 쿼리가 참조하는 파일까지 삭제될 수 있으므로 절대 운영 환경에서 테스트하지 말 것.
