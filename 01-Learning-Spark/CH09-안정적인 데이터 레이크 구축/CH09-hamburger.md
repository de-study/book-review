데이터 엔지니어링의 궁극적인 목표는 단순히 데이터를 옮기는 것이 아니라, 처리된 데이터를 쿼리하여 통찰력을 얻는 것이다.

이를 위해 최적의 스토리지 솔루션을 선택하는 것이 무엇보다 중요하다.

## 1. 스토리지 솔루션의 5가지 핵심 속성

어떤 저장소를 선택하든 다음 5가지 기준을 고려해야 한다.

-   **확장성 및 성능**: 데이터 볼륨에 따라 확장 가능하며 저지연(Low Latency)을 보장하는가?
-   **트랜잭션 지원**: 복잡한 워크로드에서도 데이터 품질(ACID)을 보장하는가?
-   **다양한 데이터 형식 지원**: 비정형(로그), 반정형(JSON), 정형(테이블) 데이터를 모두 담을 수 있는가?
-   **다양한 워크로드 지원**: SQL, 배치, 스트리밍, ML/AI 등 다양한 작업을 지원하는가?
-   **개방성**: 특정 벤더에 종속되지 않고 표준 API와 오픈 소스 형식을 사용하는가?

## 2\. DB vs 데이터 레이크

**데이터베이스 (Database)**  
특징: 엄격한 스키마, 강력한 트랜잭션(ACID) 보장, 빠른 쿼리 성능.  
한계: 데이터 크기가 테라바이트/페타바이트급으로 커지면 확장 비용이 너무 비싸고, 머신러닝 같은 비SQL 기반 분석을 지원하기 어렵다.  
  
**데이터 레이크 (Data Lake)**  
특징: 저렴한 범용 하드웨어(S3, HDFS 등)에 모든 형식의 데이터를 원본 그대로 저장. 확장성이 매우 뛰어남.  
한계: 트랜잭션 지원의 부재. 여러 명이 동시에 데이터를 쓸 때 데이터가 깨지거나(Atomicity 부족), 스키마가 일치하지 않아 데이터 품질이 저하되는 '데이터 늪(Data Swamp)' 현상이 발생할 수 있다.

## 3. 차세대 솔루션: 데이터 레이크하우스 (Lakehouse)

레이크하우스는 데이터베이스의 '데이터 관리 능력'과 데이터 레이크의 '유연성/확장성'을 결합한 새로운 패러다임입니다.  
(저렴한 스토리지 위에 데이터베이스 수준의 ACID 트랜잭션을 구현)  
주요 기능: 트랜잭션 지원, 스키마 적용 및 거버넌스, 오픈 형식 지원, 업서트(Upsert) 및 삭제 지원.

## 4\. 레이크하우스를 구현하는 오픈소스

현재 시장에서 가장 주목받는 세 가지 프로젝트. 모두 아파치 스파크와 긴밀하게 통합된다.

| 프로젝트 | 특징 |
| --- | --- |
| Apache Hudi | 우버(Uber)에서 개발. 증분 데이터 업데이트(Upsert)와 삭제에 최적화. |
| Apache Iceberg | 넷플릭스(Netflix)에서 개발. 거대한 데이터셋의 스키마 진화(페타바이트)와 '시간 탐색(Time Travel)' 기능 강점. |
| Delta Lake | Databricks에서 개발. 스파크 창시자들이 만든 만큼 스파크와 가장 완벽하게 호환되며 스트리밍 지원이 강력함. |

## 5\. 델타 레이크로 레이크하우스 구축

**델타 레이크 구축하기 (기존 테이블 변환)**

데이터 레이크의 기존 Parquet 파일을 델타 형식으로 바꾸는 것은 생각보다 간단하다.  
배치 처리: 스파크의 write 기능을 사용하여 형식을 delta로 지정하고 저장하면 끝!

```
예: df.write.format("delta").save("/data/delta-table")
```

  
스트리밍 처리: readStream과 writeStream을 활용해 실시간 데이터를 델타 테이블에 바로 적재할 수 있다.

**델타 레이크의 핵심 기능: 데이터 수정과 삭제**

-   Update & Delete: 특정 조건에 맞는 레코드를 수정하거나 삭제할 수 있다. (예: GDPR 규제에 따른 개인정보 삭제 작업)
-   Merge (Upsert): 데이터가 있으면 업데이트하고, 없으면 새로 삽입하는 복잡한 로직을 단 하나의 명령어로 처리한다.
-   Time Travel: 데이터의 과거 버전을 조회하거나 잘못된 업데이트 이전으로 복구할 수 있는 기능. (Git의 커밋 히스토리와 같은)

데이터 품질을 높이는 '멀티홉(Multi-Hop) 아키텍처'  
델타 레이크는 데이터를 단계별로 가공하며 품질을 높이는 메달리온(Medallion) 아키텍처를 권장한다.

| 단계 (Layer) | 설명 | 역할 |
| --- | --- | --- |
| Bronze (Raw) | 원천 데이터가 그대로 들어오는 곳 | 데이터 보존 및 덤프 |
| Silver (Filtered) | 중복 제거, 필터링, 스키마 정리 완료 | 분석 준비 단계 |
| Gold (Aggregated) | 비즈니스 로직 적용 및 요약 | 대시보드 및 최종 리포트용 |

## 6\. 델타 레이크의 스키마 및 관리 기법

데이터 파이프라인이 망가지는 가장 흔한 이유는 "예상치 못한 데이터 형식"이 들어올 때이다.

델타 레이크는 이를 어떻게 방지하고, 문제가 생겼을 때 어떻게 해결할까?

**스키마 강제 적용 (Schema Enforcement)**  
테이블을 생성할 때 정의한 스키마와 맞지 않는 데이터(예: 컬럼명이 다르거나 데이터 타입이 불일치하는 경우)가 들어오려고 하면 쓰기 작업을 거부하고 에러를 발생시킨다.  
이를 통해 데이터 레이크가 쓰레기 데이터로 가득 차는 '데이터 늪'이 되는 것을 원천 차단한다.  
  
**스키마 진화 (Schema Evolution)**  
비즈니스가 변하면 데이터 구조도 변하기 마련이다.

에러를 내는 대신 새로운 컬럼을 안전하게 추가하고 싶을 때는 .option("mergeSchema", "true") 설정을 사용한다.  
  
이 옵션을 쓰면 기존 데이터를 유지하면서 새로운 컬럼을 자동으로 테이블 구조에 병합한다.  
  
**테이블 상세 정보 확인 (DESCRIBE & HISTORY)**  
DESCRIBE DETAIL: 테이블의 현재 상태(파일 수, 크기, 형식 등)를 한눈에 파악한다.  
DESCRIBE HISTORY: 이 테이블에 어떤 작업(Write, Update, Delete 등)이 일어났는지 모든 이력을 조회한다. 누가, 언제, 어떤 작업으로 데이터를 변경했는지 감사(Audit)할 수 있다.  
  
**타임 트래블 (Time Travel)**  
실수로 데이터를 잘못 업데이트했다면? 델타 레이크는 과거의 특정 시점으로 데이터를 조회하거나 되돌릴 수 있다.

```
버전 기준: AS OF VERSION 5
시간 기준: AS OF '2026-04-10'
복구(Restore): RESTORE TABLE ... TO VERSION AS OF 10 명령어를 통해 테이블 전체를 특정 시점으로 즉시 복구할 수 있다.
```

  
**저장 공간 최적화: VACUUM**  
타임 트래블을 위해 델타 레이크는 이전 버전의 파일들을 보관한다.  
VACUUM 명령어를 사용하면 설정된 보존 기간(기본 7일)보다 오래된, 더 이상 사용되지 않는 파일들을 삭제하여 용량을 확보할 수 있다.  
주의: VACUUM을 실행하면 그 이전 시점으로의 타임 트래블은 불가능해지니 신중히 결정해야 한다.

**성능 최적화를 위한 팁 (Performance Tuning)**

데이터가 많아져도 속도가 느려지지 않게 관리하는 방법이다.

-   Compaction (Bin-packing): 작은 파일들이 너무 많아지면 읽기 성능이 떨어진다. OPTIMIZE 명령을 통해 작은 파일들을 적절한 크기의 큰 파일로 합쳐주어야 한다.
-   Z-Ordering: 자주 필터링 조건으로 사용되는 컬럼을 기준으로 데이터를 물리적으로 재배치하여 쿼리 성능을 비약적으로 높인다.
-   Vacuum: 더 이상 필요 없는 오래된 데이터 파일들을 삭제하여 스토리지를 정리한다. (단, 타임 트래블 기간을 고려해서 실행해야 함!)
