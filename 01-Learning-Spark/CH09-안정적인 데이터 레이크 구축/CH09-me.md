# 9. 아파치 스파크를 통한 안정적인 데이터 레이크 구축

# 1. 최적의 스토리지 솔루션의 중요성

## 1.1 확장성 및 성능

- 데이터 볼륨에 맞게 확장 가능
- 워크로드에 필요한 읽기/쓰기 처리량 및 지연 시간 제공

## 1.2 트랜잭션 지원

- 동시에 데이터를 읽고 쓰는 복잡한 워크로드에서 ACID 트랜잭션 필수

## 1.3 다양한 데이터 형식 지원

- 비정형 데이터: 텍스트 파일 등
- 반정형 데이터: json 등
- 정형 데이터: 테이블 형식 데이터 등

## 1.4 다양한 워크로드 지원

- 전통적인 BI 분석과 같은 SQL 워크로드
- 비정형 원시 데이터를 처리하는 전통적인 ETL 작업과 같은 배치 워크로드
- 실시간 모니터링 및 장애 알람과 같은 스트리밍 워크로드
- 추천 및 이탈 예측과 같은 ML 및 AI 워크로드

## 1.5 개방성

- 표준 API 사용을 통한 다양한 도구와 엔진에서 데이터 액세스

# 2. 데이터 베이스

- 데이터 베이스는 구조화된 데이터를 SQL 쿼리를 사용하여 읽을 수 있는 테이블로 저장하도록 설계
- 데이터는 데이터베이스 관리 시스템이 데이터 저장 및 처리를 모두 최적화할 수 있도록 하는 엄격한 스키마 준수 필요
- 모든 읽기/쓰기 작업에 대한 강력한 트랜잭션 ACID 보장

## 2.1 데이터베이스의 SQL 워크로드

### 2.1.1 온라인 트랜잭션 처리 (OLTP)

- 한 번에 몇 개의 레코드를 읽거나 업데이트하는 간단한 퀄로 높은 동시성, 짧은 지연 시간이 특징
- 은행 계좌 거래 등

### 2.1.2 온라인 분석 처리(OLAP)

- 많은 레코드에 대한 높은 처리량 스캔이 필요한 복잡한 쿼리(집계 및 조인 포함) 수행
- 정기 리포트 등

→ 아파치 스파크는 주로 OLAP 워크로드용으로 설계된 쿼리 엔진

## 2.2 아파치 스파크를 사용하여 데이터베이스 읽기 및 쓰기

- JDBC 드라이버가 있는 데이터베이스(PostgreSQL, MySQL)의 경우 내장 JDBC 데이터 소스를 적절한 JDBC 드라이버 jar 파일을 사용하여 데이터에 액세스 가능
- 없는 경우(애저 코스모스 DB, snowflake 등) 적절한 형식으로 호출할 수 있는 전용 커넥터 사용

## 2.3 데이터베이스의 한계

- 데이터 크기 증가:
    - 빅데이터의 도래와 함꼐 사용자 행동을 이해하기 위해 모든것을 측정하고 수집하는 추세로 데이터양이 기가바이트에서 테라바이트 및 페타바이트로 증가
- 분석의 다양성 증가:
    - 데이터 수집의 증가와 함께 더 싶은 통찰력 필요
    - 머신러닝 및 딥러닝과 같은 복잡한 분석 폭발적 성장
- 데이터베이스 확장 비용이 너무 큼:
    - 데이터 볼륨의 증가를 효율적으로 처리하기 위해 여러 시스템을 사용하여 데이터를 병렬로 처리하는 수평적 확장 필요
    - 대부분의 데이터베이스, 특히 오픈소스 데이터베이스는 분산 처리를 수행하기 위한 확장 가능한 설계가 아님
    - 소수의 산업용 데이터베이스 솔루션 이용 가능하지만 비용이 비쌈
- 데이터베이스는 비SQL 기반 분석을 잘 지원하지 않음
    - 데이터베이스는 일반적으로 해당 데이터베이스의 SQL 처리 엔진만 읽을 수 있도록 고도로 최적화된 복잡한 형식으로 데이터 저장
    - 이는 머신 러닝 및 딥러닝 시스템과 같은 다른 처리 도구가 데이터에 효울적으로 액세스할 수 없음을 의미
    - 데이터베이스는 머신러닝과 같은 비 SQL 기반 분석을 수행하도록 쉽게 확장될 수 없음

→ 위와 같은 데이터베이스의 한계로 인해 데이터 레이크가 등장

# 3. 데이터 레이크

- 데이터 레이크는 범용 하드웨어에서 실행되고, 수평으로 쉽게 확장되는 분산 스토리지 솔루션
- 데이터는 모든 처리 엔진이 표준 API를 사용하여 읽고 쓸 수 있도록 오픈 형식의 파일로 저장
- 2000년대 후반 아파치 하둡 프로젝트의 HDFS에 의해 대중화

## 3.1 데이터 레이크 구축시 선택사항

- 스토리지 시스템: HDFS 또는 클라우드 개체 저장소
- 파일 형식: 다운스트림 워크로드에 따라 정형, 반정형, 비정형 형식의 파일로 저장
- 처리 엔진: 수행할 분석 작업 부하의 종류에 따라 처리 엔진이 선택
    - 배치 처리 엔진: Spark, Presto, Hive
    - 스트림 처리 엔진: Spark, Flink
    - 머신러닝 라이브러리: Spark MLlib, scikit-learn, R

## 3.2 아파치 스파크를 사용하여 데이터 레이크에서 읽고 쓰기

- 다양한 워크로드 지원
- 다양한 파일 지원
- 다양한 파일 시스템 지원
- 주의:  클라우드 스토리지 시스템은 표준 파일 시스템에서 기대하는 것과 파일 작업의 특성이 동일하지 않은 경우가 있기 떄문에 이에 따라 스파크를 구성하지 않으면 처리 결과가 일관되지 않을 수 있음

## 3.3 데이터 레이크의 한계

데이터 레이크는 다음에 대하여 ACID 보증을 제공하지 못함

- 원자성과 독립성
    - 분산 처리에서 작업이 실패하면 이미 작성된 파일을 롤백하는 메커니즘이 없으므로 잠재적으로 손상된 데이터가 남을 수 있음
- 일관성:
    - 실패한 쓰기에 대한 원자성 부족으로 인해 데이터에 대한 일관성을 읽음
    - 성공적으로 작성된 데이터라도 데이터 품질을 보장하기 어려운 경우는 실수로 뜻하지 않게 기존 데이터와 일치하지 않는 형식 및 스키마로 데이터 파일을 저장하는 것

위 제약 사항 해결 방법

- 기존 데이터의 원자적 수정을 위해 하위 파티션 디렉터리 전체 데이터를 다시 씀
- 데이터에 대한 동시 액세스 또는 이로 인한 데이터 불일치 회피를 위해 데이터 업데이트 작업 및 데이터 쿼리 작업에 대해 일정한 시차를 두고 시행

# 4. 레이크 하우스: 스토리지 솔루션 진화의 다음 단계

레이크하우스는 OLAP 워크로드를 위한 데이터 레이크와 데이터웨어하우스의 최고의 요소를 결합한 새로운 패러다임

## 4.1 주요 기능

- 트랜잭션 지원: 동시 작업 부하가 잇는 경우 ACID 보장
- 스키마 적용 및 거버넌스
- 오픈형식의 다양한 데이터 유형 지원: 구조화, 반구조화 또는 비구조화를 포함하여 많은 새로운 데이터 애플리케이션에 필요한 모든 유형의 데이터를 저장, 정제, 분석 및 액세스 가능
- 다양한 워크로드 지원
- 업서트 및 삭제 지원: 변경 데이터 캡처(CDC) 및 저속 변경 디멘션(SCD) 작업에 레이크하우스르 사용하면 트랜잭션 보장으로 데이터를 동시에 삭제하고 업데이트 가능
- 데이터 거버넌스: 레이크하우스는 데이터 무결성에 대해 추론하고 정책 준수를 위해 모든 데이터 변경사항을 감사할 수 있는 도구를 제공 (아파치 후디, 아파치 아이스버그, 델타 레이크)
    - 세 프로젝트의 광범위한 기능:
        - 확장 가능한 파일 시스템에 정형 파일 형식으로 대용량 데이터 저장
        - 트랜잭션 로그를 유지하여 데이터에 대한 원자적 변경 타임라인 기록
        - 로그를 사용하여 테이블 데이터의 버전을 정의하고 읽기와 쓰기 간의 스냅샷 격리 보장 제공
        - 아파치 스파크를 사용하여 테이블에 대한 읽기 및 쓰기 지원

## 4.2 아파치 후디

- 우버 엔지니어링에서 처음 구축한 아파치 후디는 키/값 스타일 데이터에 대한 증분값의 업서트 및 삭제를 위해 설계된 데이터 저장 형식
- 데이터는 열 기반 형식과(parquet) 행 기반 형식(Avro)의 조합으로 저장

### 4.2.1 지원 기능

- 빠르고 플러그 가능한 인덱싱으로 업서트
- 롤백 지원을 통한 데이터 원자성 게시
- 테이블에 대한 증분 변경 읽기
- 데이터 복구를 위한 저장점
- 통계를 이용한 파일 크기 및 레이아웃 관리
- 행 및 열 데이터의 비동기 압축

## 4.3. 아파치 아이스버그

- 넷플릭스에서 구축
- 단일 타이블에서 페타바이트까지 증가할 수 있는 확장성과 스키마의 진화 특성을 가진 범용 데이터 스토리지에 중점

### 4.3.1 지원 기능

- 열, 필드 및/또는 중첩 구조의 추가, 삭제, 업데이트, 이름 변경 및 재정렬을 통한 스키마 진화 기능
- 테이블의 행에 대한 파티션값을 내부적으로 생성하는 숨겨진 파티셔닝 기능
- 데이터 볼륨 똔느 쿼리 패턴이 변경될 떄 테이블 레이아웃을 업데이트 하기 위해 메타데이터 작업을 자동으로 수행하는 파티션 진화 기능
- ID 또는 타임스탬프별로 특정 테이블 스냅샷을 쿼리할 수 있는 시간 탐색 기능
- 오류 수정을 위한 이전 버전으로의 롤백 기능
- 여러 동시 쓰기 작업 간에도 직렬화 가능한 격리 기능

## 4.4 델타 레이크

- 델타 레이크는 아파치 스파크의 창시자가 구축한 리눅스 파운데이션에서 호스팅하는 오픈소스 프로젝트
- 트랜잭션 보증을 제공하고 스키마 시행 및 발전을 가능하게 하는 오픈 데이터 저장 형식

### 4.4.1 지원 기능

- 정형화 스트리밍 소스 및 싱크를 사용하여 테이블에서 스트리밍 읽기 및 쓰기
- 자바, 스칼라 및 파이썬 API에서도 업데이트, 삭제 및 병합(업서트용) 작업
- 테이블 스키마를 명시적으로 변경하거나 데이터 프레임의 쓰기 중에도 데이터 프레임의 스키마를 테이블의 스키마에 암시적으로 병합하여 스키마를 변경 가능
- ID 또는 타임스탬프별로 특정 테이블 스냅샷을 쿼리할 수 있는 시간 탐색 기능
- 오류를 수정하기 위한 이전 버전의 롤백 기능
- SQL, 배치 처리 또는 스트리밍 작업을 수행하는 여러 동시 작성기 간의 직렬화 가능한 격리

## 4.5 주요 차이점 및 유스 케이스

### 4.5.2 주요 차이점

| **항목** | **델타 레이크** | **아파치 아이스버그** | **아파치 후디** |
| --- | --- | --- | --- |
| **핵심 설계** | 트랜잭션 로그(JSON) 중심 | 스냅샷 기반 메타데이터 관리 | 타임라인 기반 증분 처리 |
| **강점** | Spark 생태계와의 완벽한 조화 | 스키마 진화 및 파티션 숨김 기능 | 빈번한 업데이트 및 스트리밍 |
| **쓰기 최적화** | CoW(Copy on Write) | CoW(Copy on Write) | CoW 및 MoR(Merge on Read) |
| **거버넌스** | Linux Foundation | Apache Software Foundation | Apache Software Foundation |

### 4.5.3 유스케이스

- **델타 레이크 사용 시:** Databricks 환경을 사용 중이거나 Spark 중심의 파이프라인에서 빠른 성능과 안정성을 원하는 경우
- **아파치 아이스버그 사용 시:** 다양한 쿼리 엔진(Trino, Presto, Flink)을 혼합하여 사용하고 테이블 스키마가 자주 변경되거나 데이터 규모가 매우 거대한 경우
- **아파치 후디 사용 시:** 실시간에 가까운 데이터 업데이트(Upsert)가 빈번하고 스트리밍 데이터 소스를 증분 방식으로 처리해야 하는 경우

# 5. 아파치 스파크 및 델타 레이크로 레이크하우스 구축

## 5.1 델타 레이크와 아파치 스파크 구성

- 대화형 셸 설정

```python
# scala
--packages io.delta:delta-core_2.12:0.7.0

# pyspark
pyspark --packages io.delta:delta-core_2.12:0.7.0
```

- 메이븐 좌표를 사용하여 독립형 스칼라/자바 프로젝트 설정

```python
	<dependency>
	<groupId>io.delta</groupId>
	<artifactId>delta-core_2.12</artifactId>
	<version>0.7.0</version>
</dependency>
```

## 5.2 데이터 로드 및 스트리밍 처리

- **포맷 전환:** 기존 `parquet` 포맷을 `format("delta")`로 변경하는 것만으로 간편하게 마이그레이션 가능
- **배치 및 스트리밍 통합:** 동일 테이블에 대해 배치 작업과 스트리밍 작업을 동시에 수행 가능
- **다중 스트리밍 지원:** 여러 개의 스트리밍 쿼리가 동일 테이블에 데이터를 추가하더라도 'Exactly-once' 보장

## 5.3 데이터 품질 및 스키마 관리

- **스키마 강제 적용 (Schema Enforcement):** 테이블 메타데이터와 맞지 않는 데이터 형식이 입력될 경우 쓰기 작업을 차단하여 데이터 오염 방지
- **스키마 진화 (Schema Evolution):** `mergeSchema` 옵션을 사용해 새로운 컬럼을 명시적으로 추가하고 기존 데이터와의 호환성 유지

## 5.4 데이터 변환 및 DML 조작

- **ACID 트랜잭션 보장:** 업데이트(UPDATE), 삭제(DELETE), 병합(MERGE) 작업 시 데이터 일관성 유지
- **업데이트 및 삭제:** 특정 조건에 맞는 행을 수정하거나 GDPR 규제 대응을 위한 유저 데이터 삭제 기능 제공
- **업서트 (Upsert):** `merge()` 명령을 통해 기존 레코드는 업데이트하고 새로운 레코드는 삽입하는 CDC(Change Data Capture) 처리 최적화
- **중복 제거:** `INSERT-ONLY MERGE`를 활용하여 기존에 없는 데이터만 선택적으로 삽입 가능

## 5.5 변경 내역 감사 및 타임 트래블

- **작업 이력 감사 (History):** `history()` 메서드를 통해 테이블에 발생한 모든 작업의 버전, 시간, 파라미터 확인 가능
- **타임 트래블 (Time Travel):** 특정 버전(`versionAsOf`)이나 특정 시점(`timestampAsOf`)의 데이터를 조회하여 실험 재현 및 데이터 복구 수행
