# 8. 정형화 스트리밍

---

# 아파치 스파크의 스트림 처리 엔진의 진화

## 1. 스트리밍 처리 모델의 진화

데이터 스트리밍은 끝이 없는 데이터 흐름을 연속적으로 처리하는 것을 의미합니다. Spark는 이 기술을 크게 두 가지 단계를 거쳐 발전시켰습니다.

### 전통적인 방식 (Record-at-a-time)

초기 시스템들은 레코드 하나가 들어올 때마다 즉시 처리하는 방식을 사용했습니다.

- **장점:** 응답 속도(Latency)가 매우 빠릅니다(밀리초 단위).
- **단점:** 노드 장애가 발생했을 때 복구 과정이 복잡하고 효율성이 떨어집니다.

### 마이크로 배치 방식 (Micro-Batch)

Spark Streaming(DStreams)이 도입한 개념으로, 스트림을 아주 작은 시간 단위(예: 1초)의 '배치'로 나누어 처리합니다.

- **장점:** Spark의 스케줄링 시스템을 그대로 사용하여 장애 복구가 강력하고, **Exactly-once(정확히 한 번 처리)** 보장이 가능합니다.
- **단점:** 배치 단위로 모아서 처리하므로 전통적인 방식보다 지연 시간(보통 초 단위)이 발생합니다.

---

## 2. Structured Streaming의 철학

기존 DStream API의 한계(배치와 스트리밍 API의 이원화, 최적화 부족 등)를 극복하기 위해 등장했습니다. 핵심 철학은 **"스트리밍 처리도 배치 처리만큼 작성하기 쉬워야 한다"**는 것입니다.

- **무한 테이블(Unbounded Table):** 스트리밍 데이터를 끝없이 행이 추가되는 하나의 큰 테이블로 간주합니다.
- **증분 처리(Incrementalization):** 개발자가 쿼리를 작성하면, 엔진이 새 데이터만 골라내어 결과 테이블을 효율적으로 업데이트합니다.

---

## 3. 스트리밍 쿼리 정의 5단계

텍스트에서는 소켓을 통해 데이터를 읽어 단어 수를 세는 예제를 통해 5단계를 설명합니다.

1. **입력 소스 정의:** `spark.readStream`을 사용하여 데이터 소스(소켓, Kafka, 파일 등)를 설정합니다.
2. **데이터 변환:** `select`, `filter`, `groupBy` 등 일반적인 DataFrame 연산을 적용합니다. 이때 상태 유지가 필요 없는 'Stateless'와 합계/평균처럼 이전 데이터를 기억해야 하는 'Stateful' 연산으로 나뉩니다.
3. **출력 싱크 및 모드 설정:** `writeStream`을 사용하며, 출력 방식(모드)을 결정합니다.
    - **Append:** 새로 추가된 행만 출력.
    - **Update:** 변경된 행만 업데이트.
    - **Complete:** 전체 결과 테이블을 매번 출력.
4. **처리 상세 설정 (Trigger & Checkpoint):** * **Trigger:** 언제 데이터를 처리할지 결정(예: 1초마다, 혹은 한 번만).
    - **Checkpoint:** 장애 발생 시 복구를 위해 진행 상황을 저장할 경로를 지정합니다.
5. **쿼리 시작:** `.start()`를 호출하여 실제 백그라운드 스레드에서 처리를 시작합니다.

# 실행 중인 스트리밍 쿼리의 내부

## 1. 장애 복구와 Exactly-Once 보장

Structured Streaming은 장애가 발생해도 데이터를 '정확히 한 번' 처리한 것과 같은 결과를 보장합니다. 이를 위해 **체크포인트(Checkpoint)**라는 '기록 보관소'를 활용합니다.

### 복구의 핵심 요소

- **Checkpoint Location:** 쿼리의 고유 ID와 진행 상황(예: Kafka의 오프셋)을 저장합니다. 재시작 시 이 경로를 그대로 사용해야 중단된 지점부터 다시 시작할 수 있습니다.
- **Exactly-Once 성립 조건:**
    1. **재생 가능한 소스:** 장애 시 데이터를 다시 읽을 수 있어야 함 (예: Kafka, 파일).
    2. **결정적 계산:** 동일한 입력에 대해 항상 동일한 결과를 내야 함.
    3. **멱등성(Idempotent) 싱크:** 동일한 데이터를 여러 번 써도 결과가 변하지 않아야 함.

---

## 2. 활성 쿼리 모니터링

운영 환경에서는 스트리밍 쿼리가 잘 돌고 있는지 감시하는 것이 필수적입니다.

- **StreamingQuery 객체:** `lastProgress()`를 통해 처리 속도, 입력 행 수 등을 확인하고, `status()`를 통해 현재 엔진이 무엇을 하고 있는지(데이터 대기 중 등) 확인합니다.
- **주요 지표:** 특히 `processedRowsPerSecond`(초당 처리량)가 `inputRowsPerSecond`(초당 유입량)보다 낮다면 처리 지연(Backlog)이 발생하고 있다는 위험 신호입니다.
- **Listener:** Scala/Java에서는 `StreamingQueryListener`를 구현하여 쿼리의 시작, 진행, 종료 이벤트를 실시간으로 가로채어 별도의 대시보드 등으로 보낼 수 있습니다.

# 스트리밍 데이터 소스와 싱크

Spark는 기본적으로 파일, Kafka 등과 강력하게 연동됩니다.

### 파일 및 Kafka

- **파일:** 특정 디렉토리에 추가되는 파일을 스트림으로 인식합니다. 단, 파일은 원자적으로 생성되어야 하며 한 번 읽은 파일은 수정해도 다시 읽지 않습니다.
- **Kafka:** 분산 스트리밍의 표준으로, `readStream`과 `writeStream` 시 `format("kafka")`를 사용합니다. 데이터는 `key`, `value`, `topic` 등의 고정된 스키마로 들어옵니다.

### 커스텀 처리 (foreachBatch & foreach)

내장되지 않은 시스템(예: Cassandra, 외부 API)에 데이터를 쓸 때 사용합니다.

- **foreachBatch:** 마이크로 배치 결과(DataFrame)를 통째로 넘겨받습니다. 기존의 배치용 라이브러리를 그대로 재활용하거나 여러 곳에 동시에 데이터를 쓸 때 유리합니다.
- **foreach:** 레코드(행) 단위로 세밀한 로직이 필요할 때 사용하며, 연결을 열고(`open`), 쓰고(`process`), 닫는(`close`) 과정을 직접 구현합니다.
