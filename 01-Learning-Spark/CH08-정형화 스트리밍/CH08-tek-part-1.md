## 아파치 스파크의 스트림 처리 엔진의 진화
- 스트림 (stream) 처리: 끝없이 들어오는 데이터 흐름을 연속적으로 처리하는 것

**빅데이터의 등장과 함께 스트림 처리 시스템은 단일 노드 처리 엔진 → 멀티 노드 분산 처리 엔진으로 진화**
- 전통적인 분산 스트림 처리: 레코드 단위 처리 모델
	![레코드 단위 처리 모델](https://chanrankim.notion.site/image/https%3A%2F%2Fs3-us-west-2.amazonaws.com%2Fsecure.notion-static.com%2F64f88273-1c96-41f8-87c9-2cb7a68094cf%2FUntitled.png?table=block&id=5e24c49a-c1e4-4d19-a89a-dafbd86f7b18&spaceId=333f96cf-396d-45ff-8331-232d41bd4d55&width=910&userId=&cache=v2)
	- 각 노드는 지속적으로 한 번에 하나씩 레코드 받고 처리하여 생성된 레코드 다음 노드로 전송
		- 장점: 매우 짧은 응답 시간 달성 가능
		- 단점: 특정 노드가 장애 겪거나 다른 노드보다 느린 상황에서 회복하는 것에는 효과적이지 않다.
### 마이크로 배치 스트림 처리의 출현
**마이크로 배치 스트림 처리**
- 스트리밍 처리를 아주 작은 맵리듀스 스타일 배치 처리 잡 형태로써 작은 스트리밍 데이터 조각들에 대해 연속적으로 수행하는 모델
	![마이크로 배치 스트림](https://chanrankim.notion.site/image/https%3A%2F%2Fs3-us-west-2.amazonaws.com%2Fsecure.notion-static.com%2Fdbdbe37f-b590-4a62-ab99-243a4f5efdb0%2FUntitled.png?table=block&id=216d5b8f-b317-441f-914d-ed0e2389f213&spaceId=333f96cf-396d-45ff-8331-232d41bd4d55&width=1320&userId=&cache=v2)
	- 입력 스트림에서 데이터를 잘게 쪼갠다 (e.g. 1초 단위 마이크로 배치)
	- 각 배치는 분산 처리 방식으로 스파크 클러스터에서 각 마이크로 배치의 결과를 생성하는 작은 태스크 실행
	- 1초라는 배치 간격을 설정하면, 그 1초 동안 쌓인 데이터들이 하나의 덩어리가 되어 스파크 엔진으로 전달
		- 엔진은 이 덩어리를 하나의 작업으로 인식해서 빠르게 처리하고 다음 덩어리를 기다린다
		- 이 과정이 무한히 반복되면서 마치 끊임없이 데이터가 흐르는 것처럼 보이게 만드는 원리
- 원래 스파크는 대규모 데이터를 한꺼번에 처리하는 배치 엔진으로 시작
	- 이를 실시간 데이터에 적용하기 위해 데이터를 잘게 쪼개는 전략을 선택
- 가장 큰 특징은 시스템의 복잡도를 낮추면서도 높은 처리량(Throughput)을 확보할 수 있다는 점
	- 개별 데이터를 처리할 때 발생하는 오버헤드를 배치 단위로 묶어서 분산 처리함으로써 효율 극대화 
	- 하지만 필연적으로 배치 간격만큼의 지연 시간(Latency)이 발생할 수밖에 없는데, 대부분의 비즈니스 환경에서는 이 정도 지연이 큰 문제가 되지 않아 널리 쓰임

**전통적 모델 대비 스트리밍 처리를 작은 태스크로 쪼개는 것의 장점**
- 스파크의 태스크 스케쥴링은 매우 빠르고 효과적으로 태스크를 따르게 다른 Executor로 복제하여 실행<br>→ Executor 장애나 속도 저하에 대응
- 통제된 태스크의 할당으로 태스크가 여러번 실행되더라도 동일한 결과
	- 스파크 스트리밍이 전체적으로 Exactly-once를 보장할 수 있게 한다
	- 생성된 결과는 모든 입력 레코드에 대해 정확히 한 번만 처리된 결과

**마이크로 배치 모델은 밀리초 수준의 반응 속도 보여줄 수 없다**
- 초 단위 반응 속도의 단점 상쇄하는 마이크로 배치 처리의 장점
	- 왠만한 파이프라인은 굳이 초 단위 이하의 반응 속도 필요하지 않다
	- 파이프라인의 다른 부분에 더 큰 지연이 있는 경우가 많다
	- DStream API는 스파크 배치 RDD API 기반으로 작성
		- DStream은 RDD와 동일한 기능적 의미, 동일한 장애 복구 모델로 동작한다.
#### DStream (Discretized Stream)
**시간에 따라 연속적으로 들어오는 데이터를 RDD(Resilient Distributed Dataset)의 배열로 표현한 것**
- 스파크 스트리밍에서 제공하는 가장 기본적인 추상화 모델
- DStream을 다룬다 = 사실 시간에 따라 생성되는 수많은 RDD 덩어리들을 다루는 것. 
- 스파크 엔진은 이 DStream에 적용된 연산을 각 RDD에 순차적으로 적용하며 데이터 가공

**DStream은 내부적으로 시간 간격에 따라 나누어진 RDD의 연속체로 구성**
- 데이터가 시스템에 유입되면 설정된 배치 윈도우에 따라 특정 시점의 데이터들이 하나의 RDD로 묶인다
- 이 RDD는 클러스터의 여러 노드에 분산되어 병렬로 처리

**DStream 장점**
- 스파크의 핵심 장점인 장애 복구(Fault Tolerance) 기능을 스트리밍 환경에서도 그대로 누릴 수 있게 해준다. 
	- 만약 특정 노드에서 처리 중이던 데이터 조각이 소실되더라도, RDD의 계보(Lineage) 정보를 활용해 해당 시간대의 데이터만 다시 계산하면 되기 때문. 
- 별도의 복잡한 복구 메커니즘 없이도 데이터의 일관성을 유지할 수 있다는 점이 DStream이 초기 시장을 장악한 비결
- e.g. 트위터에서 1초마다 발생하는 해시태그를 집계한다고 가정
	- 1초 단위로 생성되는 RDD들이 DStream이라는 파이프라인을 타고 흐른다. 
	- 1초에 생성된 RDD에는 그 시간 동안의 트윗들이 담긴다.
	- 연산 결과는 다음 초의 RDD와 독립적으로 처리되거나 윈도우 연산을 통해 합쳐진다.

**DStream에서 데이터를 처리할 때는 단순한 변환 연산뿐만 아니라, 이전 상태를 기억해야 하는 상태 유지 연산(Stateful Operations)이 매우 중요**
- 윈도우 연산: 특정 시점부터 과거의 일정 시간 범위까지의 데이터를 묶어서 처리하는 방식 
	- DStream은 이를 슬라이딩 윈도우 형태로 지원
		- 배치 간격이 1초더라도 윈도우 크기를 10초로 설정하면, 현재 시점부터 과거 10초간의 데이터를 한 번에 분석 가능. 
		- 이는 주식 가격의 이동 평균을 구하거나 특정 시간 동안의 접속자 수를 계산할 때 필수적

**상태 관리 메커니즘은 체크포인팅(Checkpointing)이라는 기술에 의존**
- 계산된 상태 값을 주기적으로 안정적인 저장소에 저장해두는 방식
	- 만약 시스템이 멈췄다가 다시 시작되더라도 마지막으로 저장된 체크포인트부터 다시 계산을 시작할 수 있어서 데이터 유실을 방지. 
- 다만 이 과정에서 저장소 I/O 비용이 발생하므로 적절한 체크포인트 주기를 설정하는 것이 성능 최적화의 핵심
- e.g. 실제 서비스에서 사용자별 로그인 횟수를 실시간으로 카운트
	- 각 배치가 끝날 때마다 현재까지의 카운트를 상태로 저장하고 다음 배치에서 들어오는 로그인 이벤트와 합산하는 과정 필요. 
	- DStream은 이 복잡한 로직을 추상화된 API로 제공해서 개발자가 비즈니스 로직에만 집중할 수 있게 도와준다
#### DStream vs Structured Streaming
>마이크로 배치라는 철학적 개념이 먼저 정립되었고, 스파크 생태계에서 이를 처음으로 구체화하여 세상에 내놓은 기술이 바로 DStream. 
>
>시간이 흐르며 사용자들이 겪은 불편함을 해결하고 더 고도화된 추상화를 제공하기 위해 등장한 것이 지금의 표준인 Structured Streaming

그리고 시간이 흐르며 사용자들이 겪은 불편함을 해결하고 더 고도화된 추상화를 제공하기 위해 등장한 것이 지금의 표준인 Structured Streaming

**DStream**
- 데이터를 시간별로 쪼개진 조각(RDD)들의 집합으로 본다
- 데이터의 구조(Schema)를 명시적으로 다루기 어렵다
- 데이터가 시스템에 도착한 시간(Processing Time)을 기준으로 처리하는 경향이 강해서, 네트워크 지연으로 늦게 도착한 데이터를 처리하기가 까다로움

**Structured Streaming**
- 스트리밍 데이터를 끝없이 추가되는 테이블로 취급한다. 
	- 데이터가 새로 들어올 때마다 기존 테이블 아래에 행이 추가되는 개념
- SQL 쿼리를 그대로 스트리밍 데이터에 적용할 수 있고, 이벤트 시간(Event-time) 처리와 워터마크(Watermark) 같은 복잡한 개념을 훨씬 잘 해결
	- 데이터 내부에 포함된 실제 발생 시간을 기준으로 연산을 수행할 수 있는 기능 내장
- 카탈리스트(Catalyst) 최적화 엔진을 통해 쿼리 실행 속도를 비약적으로 향상
- Structured Streaming은 워터마크라는 개념을 도입해서 "특정 시간까지만 늦은 데이터를 기다려주겠다"는 기준 명시. 
	- 워터마크 설정을 통해 메모리 부하를 줄일 수 있다.

```
[ Spark Streaming Evolution Architecture ]

1. Conceptual Level: [ Continuous Data Stream ] ---> [ Micro-Batching Logic ]
    
2. Implementation 1 (DStream): 
   [ Stream ] 
   -> [ RDD t1 ] 
   -> [ RDD t2 ] 
   -> [ RDD t3 ] ... (Time-based splitting)
    
3. Implementation 2 (Structured Streaming): 
   [ Unbounded Table ] 
   | Row 1 (Event Time 10:00) | 
   | Row 2 (Event Time 10:01) | <--- New data appended as rows 
   | Row 3 (Event Time 10:02) |
```

### 스파크 스트리밍 (DStream)에서 얻은 교훈
1. 배치나 스트림 처리를 위한 단일 API 부재
	- 배치 잡을 스트리밍 잡으로 전환할 때 코드를 명시적으로 재작성 해야 했다
2. 논리/물리 계획 간에 구분 부족
	- 자동화된 최적화가 없어서 수동으로 최적화 해야 한다
3. 이벤트 타임 윈도우를 위한 기본 지원 부족
	- DStream은 각 레코드가 스파크 스트리밍에서 받은 시각을 기준으로 윈도우 연산 (Processing time, 처리 시각)
	- 실 사례에서는 레코드 받은 시각 대신 레코드 생성된 시각으로 윈도우 집게 연산 처리 필요 (Event time, 이벤트 타임)
	- 기본적으로 DStream은 이벤트 타임 윈도우를 지원하지 않는다
### 정형화 스트리밍의 철학
- 배치 및 스트림 처리를 위한 단일화된 프로그래밍 모델과 인터페이스
- 더 넓은 범위의 스트리밍 처리 지원
## 정형화 스트리밍의 프로그래밍 모델
**스트림을 일련의 무한하게 연속적으로 데이터가 추가되는 테이블의 개념으로 간주**

![Structured Streaming](https://velog.velcdn.com/images/och5351/post/f6996b8e-046c-4a6c-b257-10b4b4dc0640/image.png)
- 데이터 스트리에서 받는 모든 새로운 레코드는 무한 입력 테이블에 추가되는 새로운 행과 같다

**데이터 스트림을 테이블로 생각하는 것 →  데이터에 대한 논리적 연산 개념화 + 코드화 용이**
- 스파크의 DataFrame은 프로그래밍 방식의 테이블 표현
	- DataFrame API을 스트리밍 데이터 연산 표현하는데 쓸 수 있다
	- 스트리밍 데이터 소스로 부터 입력 DataFrame 정의 → 배치 소스에서 정의된 DataFrame에 하는 것과 같은 방식으로 DataFrame 연산 적용
### 증식화 (Incrementalization)
**스파크가 '실시간 테이블'을 완벽하게 구현해내는 핵심 기술** 
	- 배치 처리는 매번 전체 데이터를 처음부터 끝까지 다 훑지만, 스트리밍은 그러면 너무 느리리다 → 새로 들어온 부분만 영리하게 계산하는 과정

**증분화는 정적인 데이터셋에 대한 쿼리를 스트리밍 환경에서 실행 가능한 '증분 쿼리 계획(Incremental Query Plan)'으로 변환하는 과정**
- 스파크 SQL 엔진은 기본적으로 전체 데이터를 한 번에 처리하도록 설계
- 스트리밍에서는 데이터가 무한히 들어오기 때문에 매번 전체를 다시 계산하는 것은 불가능
- 엔진은 쿼리를 분석해서 이전 배치까지의 결과에 새로 들어온 데이터만 반영하여 결과를 업데이트하는 최적의 경로를 찾는다

**스파크의 쿼리 최적화 엔진인 카탈리스트(Catalyst)가 담당** 
- 카탈리스트는 사용자가 작성한 SQL이나 DataFrame 연산을 분석
	- "이 연산은 이전 상태를 저장했다가 새 데이터와 합칠 수 있는가?"를 판단
	- 만약 가능하다면, 엔진은 전체 데이터를 다시 읽지 않고도 논리적으로 동일한 결과를 내놓는 증분 실행 계획을 생성
- 시스템은 데이터가 아무리 많아져도 일정한 처리 속도를 유지할 수 있는 기반을 마련
	- e.g. 매일 아침 전교생의 출석 인원을 셀 때 처음부터 명부를 다 대조하는 게 아니라, 어제 인원에서 (+ 오늘 새로 전학 온 학생 - 자퇴한 학생) = 최종 인원을 갱신

**상태 저장소(State Store)와 증분 연산**
- 집계(Aggregation)나 조인(Join)처럼 이전 데이터를 알아야 하는 연산의 경우, 스파크는 마이크로 배치가 끝날 때마다 중간 계산 결과인 '상태'를 저장
	- 다음 배치가 시작되면 저장소에서 이 상태를 다시 불러와(Restore) 새 데이터와 합치고(Merge), 다시 저장하는(Save) 과정 반복
- e.g. 분 단위로 접속자 수를 센다고 하면, 스파크는 1분 동안의 카운트 결과를 State Store에 넣는다. 
	- 다음 10초 분량의 데이터가 들어오면, 0에서부터 세는 게 아니라 이미 저장된 카운트 값에 10초 동안 추가된 숫자만 더한다. 
	- 이런 방식은 메모리와 CPU 자원을 극적으로 아껴주지만, 상태 데이터가 너무 커지면 체크포인팅 속도가 느려질 수 있으므로 관리 필요

```
[ Query Incrementalization Workflow ]

1. User Query (DataFrame/SQL): 
   SELECT count(*) FROM stream_table GROUP BY window
    
2. Logical Plan: 
   [ Aggregate ] -> [ Filter ] -> [ Relation ]
    
3. Catalyst Incrementalization:
    
    - Identify Stateful Ops: [ Aggregate ]
        
    - Apply State Management: [ StateStoreRestore ] -> [ Aggregate ] -> [ StateStoreSave ]
        
4. Physical Execution (Micro-batch n): 
   [ New Data ] + [ Restored State (n-1) ] => [ Process ] => [ New State (n) ] & [ Result ]
```

- Structured Streaming은 이를 증분화하여 스트리밍 중 실행
### 정형화 스트리밍의 결과 모드 (Output Mode)
**스트리밍 데이터를 외부 저장소(Sink)에 어떤 방식으로 써내려갈지 결정**
#### 1. Append Mode (추가 모드)
- 가장 기본이 되는 모드로, 결과 테이블에 **새로 추가된 행**만 싱크로 내보낸다. 
- 한 번 내보낸 행은 절대 수정되거나 삭제되지 않는다는 불변성이 전제. 
	- 단순한 필터링이나 변환 연산에는 아주 적합
	- 집계(Aggregation) 연산에서는 데이터가 확정될 때까지(워터마크가 지날 때까지) 결과를 내보내지 않는다
#### 2. Update Mode (갱신 모드)
- 마지막 마이크로 배치 이후로 **변경된 행**만 골라서 내보내는 방식. 
- 기존에 있던 데이터의 값이 바뀌었거나 새로운 행이 추가되었다면 그 행들만 싱크로 전달
	- 실시간 대시보드처럼 값이 수시로 변하는 환경에서 가장 효율적. 
	- 집계 연산이 없다면 Append 모드와 똑같이 동작
#### 3. Complete Mode (전체 모드)
- 매 배치가 끝날 때마다 결과 테이블의 **전체 내용**을 다시 써버리는 방식. 
- 데이터가 아무리 많아져도 처음부터 끝까지 다 내보내기 때문에, 오직 집계 연산이 포함된 쿼리에서만 사용 가능. 
	- 전체 데이터의 스냅샷이 매번 필요할 때 유용하지만, 데이터가 무한히 쌓이면 싱크에 엄청난 부하를 줄 수 있으니 주의

|**Trigger**|**Result Table (Internal)**|**Append Sink**|**Update Sink**|**Complete Sink**|
|---|---|---|---|---|
|Batch 1|[A: 10, B: 20]|[A: 10, B: 20]|[A: 10, B: 20]|[A: 10, B: 20]|
|Batch 2|[A: 10, B: 25, C: 5]|[C: 5] (Note1)|[B: 25, C: 5]|[A: 10, B: 25, C: 5]|
|Batch 3|[A: 12, B: 25, C: 8]|[A: 12, C: 8]|[A: 12, C: 8]|[A: 12, B: 25, C: 8]|
- Note 1: Append 모드에서 집계(B의 값 변경)는 워터마크가 지나 결과가 '확정'될 때까지 출력되지 않음

|**Time Window**|**Event**|**Result Table**|**Append Sink**|**Update Sink**|
|---|---|---|---|---|
|10:00-10:10|B: 20|B: 20|(Silence)|B: 20|
|10:11-10:20|B: +5|B: 25|(Silence)|B: 25|
|10:21 (WM*)|Close|B: 25|B: 25 (OUT!)|(Silence)|
- Append 모드는 출력을 미루는 동안 그 모든 중간 데이터를 창고(State Store)에 들고 있어야 한다
	- 만약 워터마크 설정을 너무 길게 잡거나 아예 안 잡으면, 스파크는 언젠가 올지 모르는 데이터를 위해 메모리를 무한정 점유하다가 결국 터져버린다 (OOM Error)
## 정형화 스트리밍 쿼리의 기초
### 스트리밍 쿼리 정의하는 5 단계
#### 1단계: 입력 소스 지정
- 배치 데이터 소스에서 읽어 들일 때는 `DataFrameReader` 객체 만들어주는 `spark.read`사용
- 스트리밍 소스에 대해서는 `DataStreamReader` 만들어주는 `spark.readStream` 사용
#### 2단계: 데이터 변형
**상태 정보가 없는 트랜스포메이션 (Stateless Transformation)**
- 이전 상태 정보에 대한 요구가 없기 때문에 무상태 처리 가능
	- e.g. `select()`, `filter()`, `map()`
- 이러한 무상태 연산들은 배치, 스트리밍 DataFrame 양쪽에 사용 가능

**상태 정보 유지 트랜스포메이션 (Stateful Transformation)**
- 그룹화, 조인, 집계 연산과 관계되는 모든 연산들은 상태 정보 유지 트랜스 포메이션
	- e.g. `count()`
- 일부의 조합은 연산 어렵거나 증분 방식으로 계산하는 것 실행 불가능해서 지원되지 않음
#### 3단계: 출력 싱크와 모드 결정
```python
# `writeStream`은 "이제부터 이 데이터프레임을 스트리밍 방식으로 내보내겠다"는 선언
# `.format("console")`은 결과를 우리 눈에 보이는 터미널 화면(콘솔)에 출력하겠다는 뜻
# `.outputMode("complete")`은 "매 배치가 끝날 때마다 `counts` 테이블의 처음부터 끝까지 모든 내용을 콘솔에 다 찍어달라고 하는 것
writer = counts.writeStream.format("console").outputMode("complete")
```
 **DataStreamWriter**
- `writeStream` 메서드가 반환하는 객체
- 스트리밍 데이터를 외부로 내보내기 위한 모든 설정을 담는 일종의 '설계도'

 **Sink (싱크)**
- `.format("console")`처럼 데이터가 최종적으로 도착할 목적지 의미. 
- e.g. Kafka, S3, JDBC

**Append Mode: 흔적을 남기는 로그 수집**
- '새로운 행'만 내보내는 특성 때문에 상태가 없는(Stateless) 단순 변환 작업에 가장 많이 쓰인다
	- `filter()`, `select()`
> 웹 서버에서 쏟아지는 수만 개의 로그를 실시간으로 읽어서 특정 키워드만 필터링한 뒤 데이터 레이크(S3나 HDFS)에 저장하는 상황
	- 이전 데이터가 어땠는지 알 필요가 없고, 오직 지금 들어온 새로운 로그만 안전하게 저장
	- 파일 시스템은 한 번 써진 파일을 수정하는 게 비효율적이기 때문에, 새로운 데이터가 올 때마다 새 파일을 만들어 붙이는 이 방식이 아키텍처적으로 매우 안정적

**Update Mode: 변화에 민감한 실시간 대시보드**
- Update 모드는 '값이 변한 것'만 쏙쏙 골라내어 알려주는 방식
- 윈도우 집계(Windowed Aggregation) 연산과 잘 어울린다.
> 1분 단위로 현재 접속 중인 사용자 수를 카운트하는 실시간 대시보드를 만드는 상황. 
- 10시 1분에 접속자가 100명이었는데 10시 2분에 105명이 되었다면, '5명 추가'라는 변화된 결과만 대시보드에 업데이트
- `groupBy(window(...))`를 사용하여 특정 시간대별로 데이터를 묶고 그 안에서 개수를 셀 때, 배치가 돌 때마다 수치가 변하는 윈도우들만 싱크로 내보낸다. 
	- 이는 네트워크 대역폭을 아끼면서도 사용자에게는 가장 최신의 정보를 전달할 수 있는 아주 경제적인 방법
> 실무에서는 특히 카프카(Kafka)로 데이터를 다시 보낼 때 Update 모드를 애용
- 후속 시스템(예: Redis나 UI)에서 바뀐 데이터만 받아서 화면을 갱신하면 되니까 성능 최적화에 도움

**Complete Mode: 전체를 조망하는 실시간 순위표**
- 배치가 돌 때마다 '현재까지의 모든 결과'를 통째로 다시 쓰는 방식
- 주요 연산 예시는 글로벌 집계(Global Aggregation)
> 게임의 실시간 전체 랭킹이나, 서비스 시작 이후 지금까지의 누적 매출액처럼 "부분이 아니라 전체 스냅샷이 매 순간 필요한 상황
-  시간 윈도우 없이 `groupBy("user_id").sum("score")`처럼 전체 데이터를 대상으로 합계를 구할 때 자주 사용된다
> 실무에서는 Complete 모드를 쓸 때 결과 데이터의 크기가 클러스터나 싱크가 감당할 수 있는 수준인지 꼭 체크
- 사용자 수가 수천만 명인데 Complete 모드를 쓰면 매번 수천만 행을 다시 쓰느라 시스템이 뻗어버릴 수 있다.
#### 4단계: 처리 세부 사항 지정
##### 트리거 메커니즘
**트리거: 마이크로 배치 엔진이 가동되는 시간적 경계를 획정하는 논리적 프로토콜**
- `processingTime="1 second"`: 엔진에게 "매 1초마다 새로운 데이터가 있는지 확인하고, 있으면 배치를 돌려라"라고 명령
- 시스템의 지연 시간(Latency)과 처리 효율성 사이의 균형을 조절하는 핵심적인 변수

**스파크는 이전 배치가 끝난 시점부터 시간을 재는 게 아니라 고정된 시각(T, T+1s, T+2s...)을 기준으로 배치를 실행하려고 노력**
- 만약 1초 안에 처리가 안 끝나면 스파크는 다음 배치를 즉시 실행하여 지연을 최소화
- 이 방식은 데이터 유입량이 일정하지 않은 환경에서도 예측 가능한 처리 주기를 보장해주는 안정적인 장치

**트리거는 단순히 시간만 재는 것이 아니라, 하나의 배치에서 처리할 데이터의 '덩어리 크기'를 결정**
- 1초라는 짧은 트리거는 각 배치에 담기는 데이터 양을 줄여서 개별 배치를 가볍게 만들지만, 전체적인 시스템 관리 부담은 늘어난다. 
- 반대로 트리거를 10초로 늘리면 한 번에 많은 데이터를 처리하게 되어 처리량(Throughput)은 좋아지지만 지연 시간이 늘어나는 트레이드오프가 발생

##### 트리거링의 4가지 옵션
**기본 트리거 (Default/Unspecified Trigger)**
- 스파크는 '이전 마이크로 배치가 완료되자마자 다음 배치를 즉시 시작'하는 방식으로 작동. 
- 데이터가 들어오는 대로 가장 빠르게 처리하려는 성질을 가지고 있어서, 별도의 튜닝 없이도 상당히 낮은 지연 시간 갖는다

**고정 간격 트리거 (Fixed Interval Trigger)**
- 사용자가 지정한 고정된 시간 간격(예: 10초, 1분)마다 마이크로 배치를 시작하도록 강제하는 옵션
- 시스템의 박동을 규칙적으로 만들어주기 때문에 예측 가능한 운영이 가능해지고, 너무 잦은 배취 실행으로 인한 오버헤드를 줄일 수 있다.

**일회성 실행 트리거 (Once & AvailableNow Trigger)**
- 스트리밍의 탈을 쓴 배치 처리 방식. 
- `once=True`나 더 발전된 형태인 `availableNow=True`를 사용하면, 쿼리를 실행했을 때 소스에 쌓여 있는 모든 데이터를 다 처리한 뒤에 쿼리를 스스로 종료
	- 스트리밍 엔진의 강력한 오프셋 관리 기능을 쓰면서도 자원은 배치처럼 아낄 수 있다.
- 모든 처리가 끝나면 체크포인트에 기록을 남기고 안전하게 종료

**연속 처리 트리거 (Continuous Processing Trigger)**
- 마이크로 배치 방식을 아예 벗어난 모드
- `continuous='1 millisecond'`와 같이 설정하면 스파크는 데이터를 모으지 않고 들어오는 즉시 한 건씩 처리. 
- 이를 통해 지연 시간을 마이크로초(ms) 단위까지 낮출 수 있는 초저지연 모드.
- 단순한 `map`, `filter`, `select` 연산만 지원하고, `groupBy` 같은 집계 연산은 사용할 수 없다. 
- 'At-least-once' 보장만 되기 때문에 정확히 한 번 처리가 필요한 금융 데이터 등에는 쓰기 어렵다

|**트리거 옵션**|**주요 특징**|**지연 시간**|**리소스 효율성**|**추천 실무 상황**|
|---|---|---|---|---|
|**기본 (Default)**|가장 빠른 마이크로 배치|낮음|보통|빠른 테스트, 고정 유입 데이터|
|**고정 간격 (Interval)**|규칙적인 박동, 스케줄링 조절|중간|높음 (간격 길 때)|일반적 운영 환경, 처리량 최적화|
|**일회성 (AvailableNow)**|배치처럼 실행 후 자동 종료|매우 높음|최고 (비용 절감)|일간/시간 단위 배치성 스트리밍|
|**연속 처리 (Continuous)**|초저지연 (ms 단위)|매우 낮음|낮음 (자원 점유)|단순 필터링, 실시간 경고 시스템|

##### 체크포인트
**체크포인트: 시스템의 장기 기억 저장소**
- `option("checkpointLocation", checkpointDir)`: 스트리밍 쿼리가 "내가 지금 어디까지 읽었고, 중간 계산 결과는 무엇인지"를 기록할 장소를 지정
- 내부적으로 이 경로 안에는 `offsets`, `commits`, `metadata`, `sources` 같은 폴더 존재. 
	> 스파크는 매 배치가 성공할 때마다 "나 이번에 카프카 오프셋 100번까지 읽었어!"라고 기록(Commit)
	- 만약 시스템이 뻗었다가 다시 켜지면, 스파크는 제일 먼저 이 폴더를 뒤져서 "아, 마지막 기록이 100번이네. 그럼 101번부터 읽어야지!"라고 판단
	
**체크포인트의 존재 이유**
- 데이터 중복 처리 방지
> 체크포인트가 없다면, 시스템이 재시작될 때 처음부터 다시 읽거나 대충 마지막 근처부터 읽어서 데이터가 겹치거나 누락될 수 있다. 
	- 하지만 스파크는 체크포인트에 기록된 정확한 오프셋 정보를 바탕으로 소스에서 데이터를 다시 가져오기 때문에, 결과적으로 데이터가 유실되지도 않고 두 번 계산되지도 않는 상태를 만들어 낸다.
- 이 과정은 멱등성 (Idempotency)과도 관련 있다.
	- 스파크는 데이터를 출력 싱크로 보내기 전에 먼저 체크포인트에 기록을 남기려고 노력
	- 만약 기록은 성공했는데 출력하다가 죽었다면, 재시작 시 다시 출력. 
		- 이때 싱크 쪽에서 중복을 걸러줄 수 있다면 완벽한 Exactly-once가 완성

**체크포인트의 물리적 제약과 관리 주의사항**
- 체크포인트 경로는 한 번 설정하면 해당 쿼리와 평생을 함께하는 운명 공동체
	- 체크포인트 경로는 해당 디렉토리가 스트리밍 쿼리 구분하는 ID 갖고 있고 쿼리의 라이프사이클 표시하기 때문에 재시작할 때도 동일해야 한다
	- 쿼리 로직을 대대적으로 수정(e.g. 컬럼을 삭제하거나 집계 방식을 바꾸기)하면 기존 체크포인트와 충돌이 나서 쿼리가 아예 안 돌아갈 수도 있다
- 메커니즘적으로 체크포인트는 배치가 끝날 때마다 쓰기 작업을 수행
- 체크포인트가 저장되는 저장소의 쓰기 속도가 느리면 전체 스트리밍 성능이 떨어지는 병목 현상이 발생
	- S3 같은 객체 저장소는 작은 파일 쓰기에 약하기 때문에, 체크포인트가 너무 빈번하면 성능이 저하될 수 있다
- 로컬 경로는 실제 운영 환경에서는 절대 금물. 
	- 실제 서비스는 여러 대의 서버(노드)가 나누어 일을 하는데, 1번 노드가 죽어서 2번 노드가 일을 이어받으려 해도 1번 노드 하드디스크에 있는 체크포인트를 읽을 방법이 없다. 
	- 그래서 실무에서는 반드시 S3나 HDFS 같은 공유 저장소를 사용

**체크포인트를 통한 상태 보존** 
- 체크포인트 경로 안에는 단순한 오프셋뿐만 아니라 '상태(State)' 정보도 저장
	- `counts` 같은 집계 연산을 하고 있다면, 스파크는 지금까지의 카운트 값을 메모리에 들고 있다가 체크포인트 주기에 맞춰 디스크에 덤프
	- 이것이 증분 처리 가능하게 하는 물리적 기초

#### 5단계: 쿼리 시작
```python
streamingQuery = writer2.start()
```
- `start()`: 논블로킹 함수이므로 백그라운드에서 쿼리를 실행하고 호출한 즉시 객체를 리턴

```python
# 파이썬 예제
from pyspark.sql.functions import *

# 1단계: 입력 소스 지정
spark = SparkSession...
lines = (spark
         .readStream.format("socket")
         .option("host", "localhost")
         .option("port", 9999)
         .load())

# 2단계: 데이터 변형
words = lines.select(split(col("value"), "\\s").alias("word"))
counts = words.groupBy("word").count()
checkpointDir = "..."

# 3단계: 출력 싱크와 모드 결정 (전체 모드)
# 4단계: 처리 세부사항 지정
streamingQuery = (counts
                  .writeStream
                  .format("console")
                  .outputMode("complete")
                  .trigger(processingTime="1 second")
                  .option("checkpointLocation", checkpointDir)
                  .start())

# 5단계: 쿼리 시작
streamingQuery.awaitTermination()
```

## 실행 중인 스트리밍 쿼리의 내부
**쿼리가 시작되면 엔진에서는 다음 단계들을 순서대로 실행**
- 데이터프레임 연산들은 논리 계획으로 변환됨 → 스파크 SQL이 쿼리 계획을 위한 연산을 추상적으로 표현할 수 있게 함

**Structured Streaming은 내부적으로 데이터 실행 시 Spark SQL 사용**
- 스트림 처리량을 극대화하기 위해 Spark SQL의 최적화 실행 엔진의 모든 성능 끌어다 쓸 수 있다 → 성능 효율 높이는 핵심 포인트

**1. 스파크 SQL이 스트리밍 데이터에 대해 스트리밍 특성에 따라 연속적이고 효과적으로 실행할 수 있는지 논리 계획을 분석, 최적화**

**2. 스파크 SQL은 아래의 루프를 백그라운드 스레드를 통해 반복 실행**
1. 설정된 트리거 간격마다 스레드는 새 데이터가 있는지 스트리밍되는 인풋 경로(streaming source)를 확인
2. 새 데이터가 있으면 마이크로 배치로 실행
	- 최적화된 논리 계획으로부터 최적화된 스파크 실행 계획(소스로부터 새로운 데이터를 읽고, 계산을 통해 점진적으로 결과를 업데이트시키고, output에 결과를 작성하는 것까지 포함)이 생성됨
3. 모든 마이크로 배치마다 특정 분량의 데이터가 처리되고, 관련된 모든 상태가 체크포인트에 저장됨
	- 체크포인트 위치에서 쿼리가 필요할 때마다 정확한 범위를 새로 처리할 수 있게 함

**3. 위의 루프는 다음 이유들 중 하나로 질의가 종료되기 전까지 계속**
1. 질의에서 오류 발생 (처리에서 오류가 발생하거나 클러스터 장애 발생)
2. `streamingQuery.stop()` 호출에 의해 명시적으로 질의 중단
3. 트리거가 일회 실행(`once`)으로 설정되어 있는 경우
	- 질의는 모든 가능한 데이터를 처리하는 한 번의 마이크로 배치 실행 후 정지

## 스트리밍 데이터 소스와 싱크
### 파일에서 읽기
- Structured Streaming은 디렉토리에 쓰여진 파일들을 하나의 데이터 스트림으로 간주할 수 있다.
**모든 파일들은 동일한 포맷이어야 하며, 동일한 포맷을 가질 것이라고 가정**
- e.g. 포맷이 `json`이라면 모든 파일은 `JSON` 파일이어야 한다

**각각의 파일은 디렉토리에서 완전한 하나의 파일로 존재해야 한다.**
- 읽는 시점에서 전체 파일 읽기 가능해야 하고, 수정되거나 업데이트 되면 안 된다.

#### 처리해야 하는 새로운 파일 여러개 있는데, 처리량 제한 등으로 일부만 처리되는 상황
**가장 빠른 `timestamp` 갖는 파일이 먼저 선택된다.**
- 마이크로 배치 사이의 우선 순위를 의미한다
- e.g. 예를 들어 폴더에 파일이 10개가 새로 들어왔는데, `maxFilesPerTrigger`를 3으로 설정
- 스파크는 이 10개의 파일 중에서 가장 먼저 생성된(timestamp가 가장 빠른) 3개의 파일을 골라낸다.
	- 이렇게 하는 이유는 최대한 '먼저 들어온 데이터가 먼저 처리되게(FIFO)' 하기 위함
	- 이 규칙이 없다면 어제 들어온 파일이 오늘 들어온 파일보다 늦게 처리될 수 있다.
- 마이크로 배치와 마이크로 배치 사이에서는 '시간 순서'라는 질서가 엄격하게 지켜진다.
	- 1번 배치는 가장 오래된 파일들을 가져가고, 2번 배치는 그다음 오래된 파일들을 가져가는 식.

 **하지만 마이크로 배치 내에서는 따로 정의된 파일 순서가 없으며 선택된 모든 파일이 병렬적으로 처리된다.**
 - 일단 3개의 파일이 한 배치의 후보로 선정되어 공장(Executor) 안으로 들어오면, 스파크는 더 이상 파일 간의 순서를 따지지 않는다
	 - 스파크는 이 3개의 파일을 하나의 거대한 데이터 덩어리(RDD)로 합친다. 
	 - 그리고 클러스터의 여러 이그제큐터들에게 "자, 여기 데이터 조각들이 있으니까 각자 알아서 빨리 처리해!"라고 뿌려버린다
 - 스파크가 높은 처리량(Throughput)을 내는 비결. 
	 - 파일 하나하나를 순서대로 처리하면 일꾼들이 놀게 되지만, 병렬로 처리하면 모든 일꾼을 풀가동할 수 있다. 
 - 하지만 이 때문에 배치 내부에서는 데이터의 입력 순서가 뒤섞일 수 있다는 점을 명심해야 한다.

```
[ Spark File Source Processing Pipeline ]

1. Outside (Selection Phase): 
   Strict Order! 
   [File_A (10:00)] <--- Pick 1st 
   [File_B (10:01)] <--- Pick 2nd 
   [File_C (10:02)] <--- Pick 3rd --------------------------- (maxFilesPerTrigger = 3 Limit) 
   [File_D (10:03)] (Wait for next batch)
    
2. Inside Batch 1 (Processing Phase): 
   Parallel & Unordered! 
   Executor 1: Processing [Part of File_C] 
   Executor 2: Processing [Part of File_A] 
   Executor 3: Processing [Part of File_B] Executor 
   4: Processing [Part of File_C]
```

#### Structured Streaming의 추상화와 실제 실행의 차이
**Structured Streaming은 사용자에게는 데이터프레임을 제공하지만, 클러스터의 이그제큐터(Executor)가 실제로 실행할 때는 여전히 RDD를 사용**
- 스파크의 모든 고수준 API(DataFrame, Dataset, SQL)는 결국 스파크 코어(Spark Core)가 이해할 수 있는 가장 낮은 단계인 RDD로 변환되어야만 연산이 가능
>건축가(사용자)라고 생각하고 멋진 빌딩 설계도(DataFrame)를 그렸어. 
>이 설계도는 아주 깔끔하고 보기 좋지. 
>하지만 실제 공사 현장에서 일하는 인부들(Executor)은 설계도 자체를 먹을 순 없잖아? 
>설계도를 바탕으로 구체적인 벽돌 나르기, 시멘트 바르기 같은 개별 작업 지시서(RDD)가 내려와야 비로소 건물이 지어지는 거야.

- DStream은 사용자에게 "자, 여기 벽돌(RDD)이 있으니 네가 직접 쌓아봐!"라고 노골적으로 RDD를 건네주는 방식
- 반면 Structured Streaming은 "넌 설계도(DataFrame)만 그려, 벽돌 쌓는 건 내가 가장 효율적인 방법(RDD)으로 알아서 할게"라고 속이는 방식. 
	- 그래서 3개의 파일을 읽으라고 명령하면, 스파크는 내부적으로 그 파일들을 하나의 거대한 '실행 단위'로 묶고 이를 물리적인 RDD로 변환해서 이그제큐터들에게 뿌리는 것

**카탈리스트 엔진이 하는 일: 전략 수립**
- `readStream`으로 파일을 읽고 `counts`를 계산하는 코드를 실행하면, 스파크는 먼저 '논리적 실행 계획(Logical Plan)'을 세운다. 
	- 여기에는 "파일 3개를 읽어서 단어를 세겠다"는 추상적인 의도만 담겨 있다. 
	- 그 다음, 스파크는 이 의도를 실현할 수 있는 여러 가지 '물리적 실행 계획(Physical Plan)'들을 검토해. 어느 서버에서 먼저 읽을지, 데이터를 어떻게 나눌지 고민하는 단계

**텅스텐 엔진이 하는 일: 물리적 실행**
- 최종적으로 결정된 물리적 계획이 실행되는 순간, 스파크는 이를 **RDD의 집합**으로 변환. 
	- 우리가 눈으로 보는 데이터프레임의 `select`, `filter` 같은 연산들은 실제 내부적으로는 RDD의 `mapPartitions`, `compute` 같은 저수준 함수들로 치환되는 것. 

**카탈리스트 + 텅스텐 엔진이 협력한 결과물**
- Structured Streaming은 RDD를 안 쓰는 게 아니라, RDD를 **직접 다루는 불편함을 숨겨주는 것**
- 카탈리스트는 데이터프레임이라는 가면을 씌워주고, 텅스텐은 그 가면 뒤에서 RDD보다 훨씬 빠른 물리적 연산을 수행하게 만든다. 
	- 그래서 사용자는 복잡한 RDD를 직접 만지지 않아도 최고의 성능을 누릴 수 있다.

```
[ Spark Execution Hierarchy ]

Layer 1: User API (DataFrame / SQL) 
	-> "High-level, Schema-aware, Declarative"

Layer 2: Optimization Engine (Catalyst & Tungsten) 
	-> "Query optimization & Code generation"

Layer 3: Physical Execution (RDD / Spark Core) 
	-> "Low-level, Distributed, Parallel Tasks"
```

#### 파일에 쓰기
- Structured Streaming은 읽기에 쓰이는 동일한 포맷의 파일에 스트리밍 쿼리 결과를 쓸 수 있다.
	- 기존 데이터 파일 수정 쉽지 않고, 파일 새로 추가하는 것은 쉽기 때문에 추가 모드만 지원
- Structured Streaming은 디렉터리에 쓰이는 데이터 파일들의 로그를 유지하며 파일 쓰기 할 때 전체적으로 정확히 한 번 (Exactly-once) 처리를 보장. 
	- 이 로그는 `_spark_metadata`라는 하부 디렉토리에 유지
		- 이 폴더는 해당 디렉토리에 있는 수많은 파일 중 "어떤 파일이 스파크에 의해 공식적으로 인정된 파일인가"를 기록
	- 디렉토리에 스파크 쿼리가 실행될 때 일회 처리 보장 위해 (중복 읽기나 일부만 읽지 않도록) 자동적으로 로그를 써서 올바른 데이터 파일 읽어들인다.
- 재시작 사이에 결과 데이터 프레임의 스키마를 변경한다면, 결과 디렉터리의 파일들이 서로 다른 스키마를 갖고 섞여 있을 수 있다 → 디렉토리에 쿼리할 때 처리 별도 필요

### 자체 제작 스트리밍 소스와 싱크
- 스파크가 기본적으로 제공하는 카프카, 파일, 소켓 같은 소스 외에, 회사의 독자적인 프로토콜이나 특정 API로부터 데이터를 직접 당겨오고 싶을 때 자체 제작 소스 만든다

**언제 사용하는가?**
- **표준 미지원 소스:** 우리 회사만 쓰는 특수 센서 데이터망이나, 표준 프로토콜이 아닌 소켓 통신을 할 때.
- **복잡한 인증 절차:** 데이터에 접근하기 위해 매번 복잡한 토큰 갱신이나 전처리가 필요한 시스템일 때.
- **독자적인 싱크:** 데이터를 단순히 저장하는 게 아니라, 특정 API를 호출해서 Push 알림을 보내거나 제어 명령을 내려야 할 때.

**장단점**
- **장점:** 완벽하게 커스터마이징이 가능하고, 스파크의 체크포인트 시스템과 완벽하게 통합해서 일회 처리(Exactly-once)를 보장하도록 설계 가능.
    
- **단점:** 개발 난이도가 매우 높다. `Source`나 `Sink` 인터페이스를 상속받아 오프셋 관리 로직을 직접 짜야 하거든. 그래서 웬만하면 다음에 설명할 `foreachBatch`로 해결하는 경우 많다.
#### foreachBatch: 스트리밍과 배치의 가교 (가장 많이 쓰인다)
`foreachBatch`는 실무에서 **가장 강력하고 유연한 싱크 도구**. 

마이크로 배치가 생성될 때마다 그 배치의 데이터프레임을 통째로 넘겨받아서, 원하는 '어떤 로직'이든 적용 가능

**왜 그리고 언제 쓰는가?**

- **기존 배치 라이브러리 재사용:** 스트리밍을 지원하지 않는 기존의 JDBC, JDBC Sink, 혹은 사내 배치용 라이브러리를 그대로 쓰고 싶을 때.
    
- **다중 출력 (Multiple Sinks):** 하나의 스트리밍 데이터를 가공해서 하나는 S3에 저장하고, 동시에 하나는 관계형 DB에 넣고 싶을 때.
    
- **Upsert(Update + Insert) 작업:** 스트리밍 데이터프레임은 기본적으로 Append만 지원하는 경우가 많은데, `foreachBatch` 안에서는 `merge`나 `upsert` 로직을 자유롭게 짤 수 있다.

```python
# foreachBatch 실전 예시: 멀티 싱크 및 DB 업서트

def save_to_multiple_locations(df, epoch_id): 
	# 1. S3에 저장
	df.write.format("parquet").mode("append").save("s3://path/to/data") 
	# 2. Redis에 캐싱 (배치용 라이브러리 사용 가능) 
	# redis_client.save(df) 
	pass

query = streaming_df.writeStream.foreachBatch(save_to_multiple_locations).start()
```

#### foreach
`foreachBatch`가 배치 단위로 일을 한다면, `foreach`는 **데이터 한 건 한 건(Row)**에 대해 로직을 수행하고 훨씬 더 세밀한 제어가 가능

**왜 그리고 언제 쓰는가?**

- **행 단위 커스텀 로직:** 특정 데이터 값이 들어올 때만 외부 API를 호출하거나, 행마다 다른 처리를 해야 할 때.
    
- **연결 관리 최적화:** `open`, `process`, `close` 함수를 직접 구현해서 파티션별로 연결(Connection)을 열고 닫는 과정을 세밀하게 조절하고 싶을 때.

**주의점** 
- 행마다 로직이 돌아가기 때문에 성능 오버헤드가 커질 수 있다. 
- 그래서 웬만하면 `foreachBatch`를 써서 배치 단위로 묶어 처리하는 게 성능상 훨씬 유리

## 데이터 트랜스포메이션
Structured Streaming에서는 점진적 실행이 가능한 DataFrame 연산만 지원한다

### 누적 실행과 스트리밍 상태
**기억(상태)이 있기 때문에 계속해서 쌓아올리는(누적) 실행이 가능하다**

**누적 실행 (Cumulative Execution)**
- 스파크 스트리밍이 데이터를 처리할 때 매번 처음부터 다시 계산하는 게 아니라, 새로 들어온 데이터 조각(Incremental)만 처리해서 기존 결과에 더해가는 방식
- 플래너가 일회성 계획이 아닌 '연속적인 계획 묶음'을 만드는 이유
- 끊임없이 이어지는 실행이 모여 하나의 거대한 결과 테이블을 완성

**스트리밍 상태(Streaming State)**
- 누적 실행을 가능하게 만드는 '기억'. 
- 이전 배치에서 계산한 중간 결과를 어딘가에 저장해두지 않으면, 다음 배치에서 새로 들어온 데이터와 합칠 방법이 없다 
	- 이 중간 결과물이 바로 상태(State)이고, 이 상태를 유지하고 갱신하는 과정이 있기 때문에 비로소 '누적'이라는 개념이 성립

**Spark SQL의 카탈리스트 옵티마이저는 모든 DataFrame 연산들을 최적화된 논리 계획으로 변환**

**논리 계획을 어떻게 실행할지 결정하는 역할하는 Spark SQL 플래너는 연속적인 데이터 스트림 위에서 동작할 필요가 있는 스트리밍 논리 계획인지 판단**
- 상황에 다라 플래너는 논리 계획을 일회성 물리 실행 계획으로 바꾸는 대신 연속적인 실행 계획 묶음 생성
- 각각의 실행 계획은 결과 DataFrame을 누적해서 업데이트
- 실행 계획은 입력 스트림에서 들어온 새 데이터 조각만 처리하며, 가능하면 이전 실행 플랜에서 계산된 부분 결과를 추가로 처리

**각 실행은 하나의 마이크로 배치로 간주되고, 각 실행 사이에 주고받는 부분적인 중간 결과는 스트리밍 상태라고 불린다 (State)**

**DataFrame 연산들은 넓게는 누적 연산 실행 위해 상태 정보 유지 필요에 따라 무상태 / 상태 정보 유지 연산으로 분류**

```
[ 1. 분석 및 최적화 단계 (Catalyst Optimizer) ]
      DataFrame Operations (사용자 코드)
              |
              v
      Optimized Logical Plan (최적화된 논리 계획)


[ 2. 실행 계획 수립 단계 (Spark SQL Planner) ]
      Is this a Continuous Stream? (스트리밍인가?) 
              | 
              +--- [Yes] ---> 일회성 실행 대신 '연속적인 실행 계획 묶음' 생성
                               (Incremental Execution Plans)


[ 3. 마이크로 배치 실행 루프 (Micro-batch Execution) ]
      
      [ Input Source ] ----> ( New Data 조각 ) 
                                    |
                                    v
      [ State n-1 ] ------> [ 물리적 실행 계획 ] ------> [ Result Update ]
      (이전 배치의 기억)              |                 (결과 테이블 누적)
            ^                       |
            |                       v
            +-------------- [ New State n ] (배치 완료 후 저장)


[ 4. 연산 타입에 따른 차이 ]
      
      1) Stateless (무상태 연산): Filter, Select, Map
         - "기억(State)이 필요 없음" -> 그냥 들어오는 대로 처리 후 바로 출력
      
      2) Stateful (상태 유지 연산): Aggregation, Join, Window
         - "이전 기억(State n-1) 필수" -> 과거 데이터를 불러와 새 데이터와 합침
```

### 무상태 트랜스포메이션 (Stateless)
#### 입력 데이터에 대한 의존성 부재 → Stateless
**모든 프로젝션 연산(`select()`, `explode()`, `map()` 등)과 선택적 연산(`filter()`, `where()` 등)은 이전 행에 대한 정보가 없어도 입력 레코드를 개별적으로 처리 가능**
- 이러한 연산들로 이루어진 스트리밍 쿼리는 무상태 트랜스포메이션에 해당

**앞서 이미 처리된 결과 레코드는 이후의 데이터에 의해 수정할 수 없기 때문에 Append 모드로 스트리밍 싱크에 쓰여질 수는 있다.**
- e.g. 이전에 벌어졌던 `filter(price > 10000)`의 결과는 나중에 어떤 데이터가 들어와도 변하지 않기 때문
- 결과가 변하지 않는다는 확신이 있기 때문에 스파크는 기다릴 필요 없이 결과가 나오자마자 외부 저장소에 착착 쌓아버린다 (Append 방식)

**반면, 이런 쿼리들은 입력 레코드들끼리 정보를 연계하지 않기 때문에 결과에서 데이터양 크게 줄어들지 않는다**
- e.g. `map()`이나 `select()`는 1억 개의 행이 들어오면 결과도 똑같이 1억 개의 행이 나온다,
- 들어온 만큼 나오는 구조라서 결과 데이터의 부피가 입력 데이터와 비슷하게 아주 거대하다

**지속적으로 증가하는 결과 데이터는 유지 비용도 많이 들어가기 때문에 전체 모드는 지원되지 않는다**
- 무상태 연산은 데이터가 줄어들지 않고 계속 쌓인다. 
- e.g. 스트리밍이 한 달 동안 돌아서 결과가 100억 건이 쌓였다고 가정. 
	- 만약 `Complete` 모드를 지원한다면, 스파크는 1초마다 돌아가는 마이크로 배치 때마다 그 100억 건의 데이터를 매번 처음부터 끝까지 다 뽑아서 내보내야 한다.
- 그래서 무상태 연산은 데이터가 줄어들지 않으니, 스파크는 매번 전체를 보여주는 `Complete` 모드 아예 금지

### 상태 정보 유지 트랜스포메이션 (Stateful)
#### 이전 행 정보가 필요한 연산의 스트리밍 쿼리 → Stateful
- `groupby()`, `count()`

**모든 마이크로 배치에서 누적 계획은 새로운 값과 이전 배치의 값을 더하여 기록**
- 계획들 사이에서 통신이 이루어지는 부분적인 값들이 상태 정보 (State)
- 상태 정보는 스파크 Executor의 메모리에 보관되며 장애 복구를 위해 설정된 위치에 체크포인트 기록

**스파크 SQL은 올바른 결과 보장 위해 이 상태 정보의 주기를 자동적으로 관리**
#### 분산/장애 내구적인 상태 정보 관리
##### 스파크 클러스터 실행 모델: 마이크로 배치와 분산 상태 관리
- Driver와 Executor 사이에서 태스크와 상태 데이터가 어떻게 오가는지 시각적으로 정리
- 드라이버가 이그제큐터 1에게도 태스크를 주고 결과를 보고받으며, 2에게도 똑같이 태스크를 주고 보고를 받는다. 
	- 즉, 드라이버와 각 이그제큐터 사이에는 각각 **양방향 통신**이 일어난다
```
[ Spark Cluster Overview: Driver & Executors ]

+-------------------------------------------------------+
|                   Driver (머리: 관리자)                 |
|  +-------------------------------------------------+  |
|  |             Spark Scheduler (스케줄러)           |  |
|  |  1. 연산을 작은 Task로 쪼갬                        |  |
|  |  2. [ Task Queue ] 에 넣고 이그제큐터 상태 감시     |  |
|  +------------------------+------------------------+  |
+---------------------------|---------------------------+
                            |
           (Task 할당)       |       (결과/상태 보고)
           -----------------+-----------------
           |                                 |
+----------v----------+           +----------v----------+
|    Executor 1       |           |    Executor 2       |
| (팔다리: 실제 계산)   |           | (팔다리: 실제 계산)   |
+---------------------+           +---------------------+
| [ Task 실행 장소 ]   |           | [ Task 실행 장소 ]   |
| [ Memory Cache ]    |           | [ Memory Cache ]    |
+---------------------+           +---------------------+
```

##### Streaming Micro-batch: 상태 정보의 분산 처리 흐름
- 상태 정보가 포함된 스트리밍 쿼리가 각 마이크로 배치에서 어떻게 '중간 데이터(State)'를 생성하고 다음 배치로 넘기는지 과정 정리
```
[ Micro-batch n: Stateful Processing Flow ]

1. [ Source ] ----> ( New Data ) ----> [ Executor Task ]
                                            |
                                            v
2. [ State Store ] ------------------> [ 처리 로직 ]
   (이전 배치에서                             |
    캐싱된 메모리 데이터)                      |
                                            v
3. [ Sink ] <------------------------- [ 결과 생성 ]
   (외부 저장소 출력)                         |
                                            v
4. [ New State ] --------------------> [ Executor Memory ]
   (다음 배치를 위한                     (Partitioned & Cached)
    중간 데이터 생성)
```

##### 도식화의 핵심 포인트
- **드라이버의 역할 (The Brain)**: 
	- 사용자의 코드를 보고 "이번 배치는 100개의 태스크로 나누자!"라고 결정해서 큐에 던진다. 
	- 드라이버는 직접 데이터를 만지지는 않지만, 누가 어떤 일을 할지 완벽하게 통제.
    
- **이그제큐터의 메모리 활용 (The Muscle & Memory)**: 
	- 각 이그제큐터는 자기가 맡은 파티션의 '상태(기존 기억)'를 자기 메모리에 들고 있다. 
	- 그래서 다음 배치가 시작될 때 멀리서 데이터를 가져올 필요 없이, 자기 메모리에서 바로 꺼내서 새 데이터와 합칠 수 있다. 
	- 이것이 바로 스파크 스트리밍이 빠른 이유
    
- **상태 데이터의 분산과 효율성**: 
	- 완전하게 파티션된 상태라는 말은, 1번 이그제큐터가 가진 기억과 2번 이그제큐터가 가진 기억이 서로 겹치지 않게 아주 효율적으로 나누어져 있다는 뜻. 
	- 덕분에 이그제큐터가 늘어나도(Scale-out) 기억 장치가 함께 늘어나서 성능이 유지될 수 있어.

![정형화 스트리밍에서 분산 상태 관리](https://chanrankim.notion.site/image/https%3A%2F%2Fs3-us-west-2.amazonaws.com%2Fsecure.notion-static.com%2Fe541b99c-2ba6-4785-9f02-5b65590eb01c%2FUntitled.png?table=block&id=03a5d286-97a5-4e0d-9de1-c297ed26d9cd&spaceId=333f96cf-396d-45ff-8331-232d41bd4d55&width=1410&userId=&cache=v2)

**1. 각 마이크로 배치는 새로운 단어를 읽어 들인다.** 
- 이그제큐터는 각 배치마다 새로 들어온 단어들을 포함하여 셔플 및 그룹화하고, 개수를 세어서 새로운 개수를 출력한다. 
- 또한 다음 마이크로배치를 위해 상태 정보를 이그제큐터의 메모리에 캐싱한다.

**2. 다음 마이크로 배치 데이터는 직전 마이크로 배치와 동일한 작업을 하고, 새로운 개수를 상태 정보의 기존 개수에 더해서 상태 정보를 업데이트한다.**

**3. 장애 발생 시 메모리의 정보를 잃을 위험이 있기 때문에 각 배치에 변화된 변경 내용과 오프셋 범위로 체크포인트 로그를 남긴다.**
- 상태 정보 유지 + 체크포인트 로그를 통해 장애 발생 시에 해당 배치 이전의 동일한 상태 정보로 동일한 입력 데이터를 재처리
- 이는 정확히 한 번 (Exactly-once) 실행을 보장하기 위한 핵심적인 요소
#### 상태 정보 유지 연산의 종류
**스트리밍 상태 유지 정보의 본질은 과거 데이터의 요약 정보를 유지하는 것**
- 오래 지난 요약 정보는 새로운 요약 정보를 위한 공간 만들기 위한 목적으로 정리되어야 할 필요 있다.
##### 관리형 상태 정보 유지 연산 
**어떤 연산이 만료 인지 아닌지를 판단해 자동적으로 오래된 상태 정보를 감지하고 정리**
- 스트리밍 집계 연산
- 스트림 - 스트림 조인
- 스트리밍 중복 제거
