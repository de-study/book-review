## 1. 스트리밍 데이터 소스 및 싱크 (Data Sources and Sinks)

### 1-1. 파일 시스템 (Files)
* **읽기 (Reading)**
    * 특정 디렉터리에 새로 추가되는 파일들을 데이터 스트림으로 인식함
    * **스키마 지정 필수**: 스트리밍 특성상 스키마 추론이 아닌 명시적 정의가 필요함
    * **원자성 보장**: 파일은 쓰기가 완전히 완료된 상태로 나타나야 하며, 이후 수정은 불가능함
    * **처리 순서**: 타임스탬프가 빠른 순서로 읽으며, 배치 내에서는 병렬로 처리함
* **쓰기 (Writing)**
    * **Append 모드만 지원**: 기존 파일 수정이 어려워 새로운 파일을 추가하는 방식만 가능함
    * **Exactly-once 보장**: `_spark_metadata` 로그를 통해 중복 처리나 부분 쓰기를 방지함

### 1-2. 아파치 카프카 (Apache Kafka)
* **읽기 (Reading)**
    * `format("kafka")`를 사용하며 브로커 주소와 구독할 토픽 설정이 필요함
    * **고정 스키마**: 결과 데이터프레임은 `key`, `value`, `topic`, `partition`, `offset`, `timestamp` 등의 컬럼을 가짐
* **쓰기 (Writing)**
    * 출력 데이터프레임에 `value`(필수) 및 `key`(선택) 컬럼이 포함되어야 함
    * 상황에 따라 `Update` 모드 등 다양한 출력 모드 선택이 가능함

### 1-3. 사용자 정의 소스 및 싱크 (Custom Logic)
* **foreachBatch()**
    * 마이크로 배치별 출력 데이터프레임에 대해 임의의 함수 실행함
    * **장점**: 기존 배치 데이터 소스 재사용 가능, Multi-sink 가능, 스트리밍 미지원 API 적용 가능함
    * **주의**: 중복 실행 시 `batchId`를 활용한 중복 제거 로직이 필요함 (At-least-once)
* **foreach()**
    * 행(Row) 단위로 커스텀 로직을 적용함
    * `open()`, `process()`, `close()` 단계를 구현하여 세밀한 연결 관리가 가능함

## 2. 데이터 변환 (Data Transformations)

### 2-1. 데이터 변환의 분류
* **무상태 변환 (Stateless Transformations)**
    * 이전 데이터와 상관없이 각 레코드를 독립적으로 처리함
    * 예시: `select()`, `filter()`, `map()`, `flatMap()`, `explode()`
* **상태 유지 변환 (Stateful Transformations)**
    * 이전 마이크로 배치의 중간 결과(상태)를 유지하고 다음 배치와 결합함
    * 예시: `count()`, `groupBy()`, `join()`, `window()`

### 2-2. 상태 관리 방식
* **관리형 상태 연산**: 스파크가 자동으로 오래된 상태를 식별하고 정리함 (예: 스트리밍 집계, 중복 제거)
* **비관리형 상태 연산**: 사용자가 직접 상태 정리 로직 정의함 (예: `mapGroupsWithState`)

### 2-3. 이벤트 시간 기반 윈도우 집계
* **텀블링 윈도우 (Tumbling Windows)**: 겹치지 않는 고정 크기의 시간 창임
* **슬라이딩 윈도우 (Sliding Windows)**: 일정 간격으로 겹쳐서 생성되는 시간 창임
* **특징**: 늦게 도착한 데이터(Late Data)도 이벤트 시간 기준으로 적절한 윈도우에 할당됨

### 2-4. 워터마크 (Watermarking)
* **정의**: 현재까지 관찰된 최대 이벤트 시간에서 특정 지연 허용치를 뺀 임계점임
* **역할**: 워터마크 이전의 윈도우를 확정된 것으로 간주하고 상태에서 삭제하여 메모리를 관리함
* **제약**: `groupBy` 이전에 `withWatermark()` 호출이 필수이며, 윈도우와 동일한 시간 컬럼을 사용해야 함

### 2-5. 출력 모드와 상태 정리의 관계
| 출력 모드 | 상태 정리 (Cleanup) 여부 | 특징 |
| :--- | :--- | :--- |
| **Update** | 워터마크에 의해 정리됨 | 변경된 행만 출력하여 가장 효율적임 |
| **Complete** | 정리되지 않음 | 모든 결과를 출력하므로 메모리 누적 주의 필요함 |
| **Append** | 워터마크 통과 시 정리됨 | 최종 확정값만 출력하며 워터마크만큼 출력 지연 발생함 |

## 3. 스트리밍 조인 (Streaming Joins)

### 3-1. 스트림-정적 데이터 조인 (Stream-Static Joins)
* 무상태 연산으로 워터마크가 불필요함
* **범위**: Inner Join 및 스트림이 주체인 Outer Join(Left/Right) 지원함
* **팁**: 정적 데이터 `cache()` 권장, 소스 변경 시 쿼리 재시작 필요할 수 있음

### 3-2. 스트림-스트림 조인 (Stream-Stream Joins)
* **상태 관리**: 매칭을 위해 양쪽 스트림 데이터를 상태로 버퍼링함
* **필수 설정 (상태 정리용)**:
    1. **Watermark**: 양쪽 입력에 대한 지연 허용 범위 설정함
    2. **시간 범위 조건**: 이벤트 간 시간 차이에 대한 제약 조건 명시함
* **외부 조인**: 워터마크가 필수이며, 매칭 실패 확신 시점까지 결과 출력이 지연됨

### 3-3. 임의의 상태 유지 연산 (Arbitrary Stateful Computations)
* **주요 API**: `mapGroupsWithState`(1:1 결과), `flatMapGroupsWithState`(1:N 결과)
* **타임아웃 (Timeouts)**
    * **Processing-time**: 시스템 시각 기준이며 구현 용이함
    * **Event-time**: 데이터 시간 및 워터마크 기준이며 지연 상황에 강인함

## 4. 고급 연산 및 성능 최적화 (Advanced Operations & Tuning)

### 4-1. flatMapGroupsWithState()의 활용
* `mapGroupsWithState()`의 유연성 한계 극복함
* **주요 장점**:
    * **임의의 출력 개수**: Iterator 반환을 통해 배치당 0개~여러 개 레코드 출력 가능함
    * **출력 모드 설정**: `Append`(파일 저장 가능) 또는 `Update` 모드 명시 가능함

### 4-2. 성능 튜닝 (Performance Tuning)
* **클러스터 자원 할당**: 무상태 쿼리는 CPU 중심, 상태 유지 쿼리는 메모리 중심 할당함
* **셔플 파티션 수 조절**: 기본값(200)은 너무 큼. 전체 코어 수의 **2~3배** 수준으로 축소 권장함
* **소스 속도 제한 (Rate Limits)**: `maxFilesPerTrigger` 등을 통해 트래픽 급증 시 안정성 확보함
* **다중 쿼리 관리**: `spark.scheduler.pool` 설정을 통해 단일 드라이버 내 쿼리 간 자원 배분을 최적화함
